# 05 · 后端接口设计

> 风格：**RESTful + 资源化路径**，少量复杂动作（报名、抢位、退款）用动词子资源。
> 协议：HTTPS / HTTP/2；JSON Body；时间字段统一 ISO 8601 UTC。
> 主要 BFF：`/api/client/v1/*`（C 端）、`/api/admin/v1/*`（B 端后台）、`/api/open/v1/*`（开放/三方回调）。
> NestJS / PostgreSQL 具体实现细节见 [10-技术实施细则](./10-技术实施细则.md)。

## 1. 接口规范

### 1.1 统一响应结构
```json
{
  "code": 0,
  "msg": "ok",
  "data": { ... },
  "trace_id": "01HXY..."
}
```
- `code = 0` 表示成功。
- 业务错误：`code != 0`，`msg` 为可展示给用户的文本。
- 列表：
```json
{
  "code": 0,
  "msg": "ok",
  "data": {
    "list": [...],
    "total": 1234,
    "cursor": "eyJpZCI6MTAwfQ==",
    "has_more": true
  }
}
```

### 1.2 请求约定
| 项 | 约定 |
| --- | --- |
| Content-Type | `application/json; charset=utf-8` |
| 分页 | 默认 cursor 分页（`cursor` + `limit`，最大 100）；后台兼容 page/page_size |
| 排序 | `sort_by=created_at&order=desc` |
| 多语言 | `Accept-Language: zh-CN` |
| 幂等 | 写接口需 `Idempotency-Key`（UUID），服务端 60s 内去重 |
| 鉴权 | `Authorization: Bearer <jwt>` |
| 多租户 | `X-Tenant-Id`（C 端）；后台 JWT 中已携带 |

### 1.3 错误码（节选）
| code | 含义 | HTTP |
| ---- | ---- | ---- |
| 0 | 成功 | 200 |
| 40001 | 参数错误 | 400 |
| 40101 | 未登录 | 401 |
| 40301 | 无权限 | 403 |
| 40401 | 资源不存在 | 404 |
| 40901 | 状态冲突（活动已结束等） | 409 |
| 42901 | 限流 | 429 |
| 43001 | 库存/席位不足 | 409 |
| 43002 | 钱包余额不足 | 409 |
| 43003 | 重复报名 | 409 |
| 50001 | 服务异常 | 500 |
| 50301 | 下游不可用 | 503 |

### 1.4 鉴权流程（C 端微信小程序）
```
1. wx.login() → code
2. POST /api/client/v1/auth/wx-login {code, tenant_id}
   ← {access_token, refresh_token, expires_in, user_id, is_member}
3. 后续请求 Authorization: Bearer <access_token>
4. 401 → POST /auth/refresh {refresh_token} 续签
```

### 1.5 鉴权流程（后台）
```
1. POST /api/admin/v1/auth/login {username, password, captcha}
2. 返回 challenge_id（如开启 2FA）
3. POST /api/admin/v1/auth/verify {challenge_id, sms_code}
4. 返回 access_token, refresh_token, menu_tree
```

### 1.6 鉴权（聚合中间件）
```
[Gateway]
  ├─ JWT 验签 → 解出 user_id / tenant_id / role_codes
  ├─ 注入 ctx 上下文：trace_id, user, tenant
  └─ 按路径路由到 client-api / admin-api / open-api
[API Runtime]
  ├─ Permission Guard（按 path / metadata → permission code 校验）
  ├─ RateLimit Guard（per-user / per-tenant）
  └─ Idempotency Interceptor
```

### 1.7 多租户越权防御（高风险，必须执行）

> SaaS 系统最常见、最严重的安全漏洞是「跨租户越权」。任何能让攻击者读到/写到不属于自己俱乐部数据的路径都是 **P0 漏洞**。本节是 Gate-keeping 清单。

#### 1.7.1 三道闸门

| # | 闸门 | 实施位置 | 失败动作 |
| - | --- | --- | --- |
| 1 | **`X-Tenant-Id` 必须由服务端校验** | `TenantGuard` / `TenantContextInterceptor` | 攻击者伪造 header 切到别家俱乐部 → 校验 `members WHERE tenant_id=? AND user_id=?` 不存在则返回 `40301 NOT_TENANT_MEMBER` |
| 2 | **所有 repository 必须 tenant-aware** | `tenantScope(ctx)` / `repo.withTenant(ctx)` | 任何 `SELECT/UPDATE/DELETE` 都必须显式携带 `tenant_id`；平台超级管理员需显式 `withSystemScope(reason)` 才能跳过，并写审计日志 |
| 3 | **资源属主二次校验** | service 层 | 即使查询拿到了行，handler 还要二次断言「该 `activity_id` 真的属于当前 tenant」「该 `registration_id` 真的属于当前 user_id」，避免遍历 ID 攻击 |

#### 1.7.2 必须遵守的编码规范

```ts
// ❌ 反例：直接使用前端传入的 tenant_id
@Get(':id')
async get(@Param('id') id: number, @Headers('x-tenant-id') tenantId: string) {
  return this.activityRepo.findByIdAndTenant(Number(id), Number(tenantId));
}

// ✅ 正例：tenant_id 只能来自服务端已解析的请求上下文
@UseGuards(JwtAuthGuard, TenantGuard)
@Get(':id')
async get(@Param('id') id: number, @Req() req: RequestWithContext) {
  const ctx = req.ctx;
  const activity = await this.activityRepo.getRequired(ctx, Number(id));

  if (activity.tenantId !== ctx.tenantId) {
    throw new ForbiddenException('CROSS_TENANT_ACCESS');
  }

  return ActivityDto.from(activity);
}
```

- **禁止** route handler 或 header 解析层把前端传来的 `tenant_id` 直接用于查询；
- **禁止** repository 里出现只按 `id` 查业务表、不带 `tenant_id` 的实现；
- **禁止** 把别人的 `registration_id / order_no / wallet_id` 传进来就照单全收 —— 必须断言属主。

#### 1.7.3 自动化测试（必须有）

每个写接口在 CI 中跑 `cross_tenant_test.yml` 矩阵：

| 场景 | 期望结果 |
| --- | --- |
| 用户 A（tenant=1）持自己 token，伪造 `X-Tenant-Id: 2` 调任意接口 | `40301 NOT_TENANT_MEMBER` |
| 用户 A 调 `GET /activities/{B 俱乐部的 activity_id}` | `40401 NOT_FOUND`（不能返回 403 暴露存在性） |
| 用户 A 调 `DELETE /registrations/{属于用户 B 的 reg_id}` | `40301 NOT_OWNER` |
| 平台管理员未带 `withSystemScope(reason)` 的查询打到全局数据 | repository helper 必须拒绝无租户上下文的业务查询 |
| 一个故意不带 tenant_id 的 repository 查询（系统越权测试用） | 强制写审计日志（`audit_logs` 中可查到该次操作） |

#### 1.7.4 平台账号切租户

平台超级管理员可在后台「切换租户视角」调试租户数据，必须做到：

- 切换动作发短信 / 邮件二次确认；
- 切换后所有 `audit_logs` 行写入 `operator_type='platform_impersonate'`，操作和真实主理人区分；
- 切换不能用于资金类写操作（充值 / 退款 / 余额调整 / 结算），这些操作只允许俱乐部本人账号或经审批的人工通道。

#### 实现要点（NestJS 伪代码）
```ts
@Injectable()
export class RegistrationService {
  async register(ctx: RequestContext, cmd: RegisterActivityCommand) {
    return this.idempotencyService.execute(ctx, cmd.idempotencyKey, 60, async () => {
      const activity = await this.activityRepo.getRequired(ctx, cmd.activityId);
      this.activityDomain.ensureRegistering(activity, ctx.userId, ctx.now);

      const seat = await this.seatGateway.grabSeat({
        activityId: activity.id,
        userId: ctx.userId,
        count: 1 + cmd.extraCount,
      });

      try {
        return await this.prisma.$transaction(async (tx) => {
          const reg = await this.registrationRepo.insert(tx, ctx, {
            activity,
            command: cmd,
            seat,
          });

          await this.registrationRepo.insertParticipants(tx, ctx, reg.id, cmd.participants);

          if (cmd.shareToken) {
            await this.registrationRepo.incrementShareRegisterCount(tx, ctx, cmd.shareToken);
          }

          let payParams: ClientPayParams | null = null;

          if (cmd.payMethod === 'wallet') {
            await this.walletService.debitForActivity(tx, ctx, reg.userId, reg.payAmount, reg.regNo);
            await this.registrationRepo.markPaid(tx, ctx, reg.id);
          } else {
            const order = await this.paymentService.createActivityOrder(tx, ctx, reg, cmd.payMethod);
            payParams = await this.paymentService.buildClientParams(ctx, order);
          }

          await this.outboxRepo.append(tx, {
            tenantId: ctx.tenantId,
            eventType: 'activity.registered',
            aggregateType: 'registration',
            aggregateId: reg.id,
            payload: { registrationId: reg.id, activityId: activity.id },
          });

          return { ...reg, seatNo: seat.seatNo, payParams };
        });
      } catch (error) {
        await this.seatGateway.releaseSeatQuietly({
          activityId: activity.id,
          userId: ctx.userId,
          count: 1 + cmd.extraCount,
        });
        throw error;
      }
    });
  }
}
```

#### Redis Lua（GrabSeat 简化）
```lua
local left_key  = KEYS[1]    -- activity:{id}:left
local set_key   = KEYS[2]    -- activity:{id}:participants
local user_id   = ARGV[1]
local extra     = tonumber(ARGV[2])

if redis.call('SISMEMBER', set_key, user_id) == 1 then
    return {-3}  -- ALREADY_REGISTERED
end

local need = 1 + extra
local left = tonumber(redis.call('GET', left_key) or '0')
if left < need then
    return {-1}  -- OUT_OF_SEATS
end

redis.call('DECRBY', left_key, need)
redis.call('SADD',   set_key, user_id)
return {0, left - need}
```

### 4.5 支付回调（开放）
```
POST /api/open/v1/payments/wechat/notify

Body: 微信支付原始报文（XML/JSON）

逻辑：
1) 验签
2) 查询 payment_orders by out_trade_no（SELECT ... FOR UPDATE）
3) 已是 paid 直接返回成功（幂等）
4) 命中 canceled / expired 状态：见下方「迟到回调处理」
5) 状态机推进：created/paying → paid，写 paid_at
6) 触发对应 biz_type 处理（同事务）：
   recharge → 钱包入账（cash/gift），记 wallet_transaction（含业务幂等键）
   activity → registration.MarkPaid + 候补/席位确认 + 事件
   mall     → 订单 paid + 扣库存 + 通知
7) 写 outbox 事件（异步发订阅消息 / WS 推送）
8) 返回 success
```

#### 4.5.1 重支付（用户超时未付重新发起）

```
POST /api/client/v1/recharge-orders/{id}/pay     # 充值的重支付
POST /api/client/v1/activities/{id}/registrations/{regId}/pay   # 活动的重支付
POST /api/client/v1/mall/orders/{no}/pay         # 商城的重支付
```

服务端实现（统一在 `PaymentService.createOrRetry`）：

1. `SELECT ... FOR UPDATE` 锁该业务单 (`biz_type`, `biz_order_no`) 全部 `payment_orders`。
2. 如已存在 `paid` 行 → 直接返回「已支付」错误。
3. 把所有 `created/paying` 行 `UPDATE status='canceled'`（幂等，可能为空）。
4. 新建一行 `attempt_no = MAX(attempt_no) + 1`，**重新生成 `out_trade_no`**，调用三方下单。
5. 返回新的 `pay_params` 给前端。
6. 旧支付单的迟到回调 → 见 §4.5.2。

#### 4.5.2 迟到回调处理

回调到达时业务单已经被切到 `attempt_no = N+1`：

| 旧单状态 | 新单状态 | 处理 |
| --- | --- | --- |
| `canceled` / `expired` | `created` / `paying` | **旧单回调代表用户实际付了旧单**：把旧单 `paid`，触发履约；同时把新单 `canceled`（应用层调三方关单 API 释放预下单） |
| `canceled` | `paid` | 用户重复付了两次：旧单按"重复支付"流程退款（异步原路退回） + 写 `audit_logs` 高优告警 |
| 其他组合 | — | 全部按"丢弃 + 记日志"处理，并发场景留给定时对账兜底 |

> 服务端禁止仅凭"旧单已 canceled"就丢弃回调；丢弃前必须确认对应业务单已被新尝试覆盖且新尝试已成功。

### 4.7 席位实时同步

#### 4.7.1 MVP 阶段：短轮询（默认）

```
GET /api/client/v1/activities/{id}/seats?since=<version>

Response:
{
  "code": 0,
  "data": {
    "version": 87,                    // 单调递增，前端用作 since
    "join_count": 12,
    "left": 4,
    "waitlist_count": 2,
    "joined_user_ids": [1001, 1002],   // 可选，按隐私规则脱敏
    "stale": false                     // 缓存兜底时为 true
  }
}
```

- 前端进入活动详情后 3-5s 轮询；列表页不轮询，只在进入详情时拉一次。
- 服务端响应来自 Redis（`activity:{id}:seats`，TTL 短，每次报名/取消时主动失效），不走 DB。
- `version` 由服务端在每次席位变化时 `INCR`；客户端 `since == version` 时返回 `not_modified` 体（响应只含 version），减少带宽。

#### 4.7.2 M2/M4 阶段：WebSocket 增量推送（规划）

```
ws://host/ws/activities/{id}?token=...

  "type": "seat_update",
  "data": {
    "join_count": 12,
    "left": 4,

  }
```

**启用条件**：
- 单活动峰值并发 > 200 在线，或日报名 PV > 50w；

- 部署侧具备 ws-gateway 独立水平扩展能力（避免与 BFF 共池被慢连接拖死）。

> 启用后，短轮询保留作为兜底：客户端连接失败 / 心跳超时自动降级，业务功能不受影响。

## 6. 并发与一致性策略汇总

| 场景 | 风险 | 解决 |
| --- | --- | --- |
| 多人同时抢最后一个席位 | 超卖 | Redis Lua 原子 + DB 活跃报名唯一键 + seat 唯一键 |
| 用户同时点两次报名 | 重复报名 | Idempotency-Key + DB 唯一索引 |
| 钱包并发扣款 | 余额负数 | 乐观锁 version + 重试；失败回滚释放席位 |
| 支付回调重复 | 重复入账 | payment_orders 状态机 + 行锁；biz_type 处理幂等 |
| 商城同款抢购 | 超卖 | Redis 库存预扣 + 异步落库；DB stock - locked_stock 兜底 |
| 排期块二次占用 | 串号 | court_schedules.uk(court_id, play_date, start_time) |
| 包场 5min 占位过期 | 占用未释放 | 延时任务 / MQ 延时消息 5min；定时任务兜底 |
| 资金账目对不齐 | 账目偏差 | 钱包流水、支付单、退款单、结算单分层核对；每日对账任务 + 告警 |
| 热点活动流量瞬时冲高 | API 被打满 / Redis 热键 | 活动级动态限流 + 席位快照缓存 + API 多副本扩容 |
| 支付回调洪峰 | 回调堆积 / 主链路被挤占 | `open-api` 独立部署 + 独立限流 + 回调先落库后异步扇出 |

## 7. 限流与熔断

- **网关层**：云 WAF / API Gateway / Ingress 按 token / IP 做基础限流，并把 `/api/client/*`、`/api/admin/*`、`/api/open/*` 路由到不同 upstream。
- **应用层**：NestJS Guard / Interceptor + Redis 令牌桶实现，推荐 `rate-limiter-flexible`，按 `(tenant_id, user_id, api)` 维度限流；登录、报名、支付下单、回调等热点接口单独配置。
- **热点活动保护**：对单个 `activity_id` 配动态阈值；活动详情与席位接口支持缓存和快速失败，避免洪峰直接穿透数据库。
- **下游熔断**：统一 HTTP Client 封装超时、重试、指数退避和熔断；如微信支付下游异常 5 次/10s → 熔断 30s 拒绝新请求并提示稍后再试。
- **降级**：
  - 排期实时查询失败 → 返回缓存（最近一次）+ 标志 `stale: true`。
  - 商城列表异常 → 先回退到 PostgreSQL 查询。

## 9. 通用 Guard / Interceptor / Filter 清单（NestJS）

> NestJS 中通常拆为：`Guard`（鉴权/权限）+ `Interceptor`（trace/响应/审计）+ `Filter`（异常统一收口）+ `Pipe`（DTO 校验）+ `Decorator`（上下文注入）。命名按 `XxxGuard / XxxInterceptor / XxxFilter / XxxPipe` 规范。

| 组件 | 类型 | 作用 |
| --- | --- | --- |
| `AppExceptionFilter` | Filter | 统一捕获业务异常 + 系统异常 → 标准 `{code, msg, data, trace_id}`；5xx 兜底脱敏 |
| `TraceInterceptor` | Interceptor | trace_id 注入 / W3C TraceContext 透传，写入日志上下文与响应头 |
| `AccessLogInterceptor` | Interceptor | 结构化访问日志（含耗时、状态、tenant_id、user_id），慢请求阈值标记 |
| `JwtAuthGuard` | Guard | JWT 解析、黑名单校验、注入 `req.user` |
| `TenantGuard` | Guard | 解析 `X-Tenant-Id` 或 JWT 中的 tenant，写入 `req.ctx.tenantId`；**强制校验当前 user 在该 tenant 下有 member 关系** |
| `PermissionGuard` | Guard | 路径 / 路由元数据 → permission_code 校验 |
| `RateLimitGuard` | Guard | Redis 令牌桶，按 `(tenant_id, user_id, api)` 维度 |
| `IdempotencyInterceptor` | Interceptor | 写接口 `Idempotency-Key` 校验 + 结果回放 |
| `AuditInterceptor` | Interceptor | 后台写操作落 `audit_logs` |
| `LocaleInterceptor` | Interceptor | `Accept-Language` → locale 注入 |
| `tenantScope(ctx)` | repository helper | repository 默认拼 `tenant_id`；平台级查询需显式 `withSystemScope(reason)` 跳过 |
