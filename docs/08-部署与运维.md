# 08 · 部署与运维

## 1. 环境分层

| 环境 | 用途 | 数据 |
| --- | --- | --- |
| dev | 本地开发、自测 | 本地 Docker |
| sit | 联调与回调验证 | 仿真测试数据 |
| staging | 准生产、压测、容灾演练 | 与生产同结构、脱敏数据 |
| prod | 生产 | 真实数据 |

所有环境必须：

- 独立 PostgreSQL
- 独立 Redis
- 独立对象存储配置
- 独立域名和密钥

禁止跨环境调用。

## 2. 部署原则

### 2.1 总原则

1. **开发单仓，生产多运行时**
2. **生产默认高可用**
3. **无状态 API**
4. **核心流量隔离**
5. **托管优先**
6. **回滚和补偿优先**

### 2.2 为什么不能按“小项目轻部署”做

目标客户是年营收过亿的羽毛球业务公司，风险点不是“功能不够多”，而是：

- 热点活动开抢洪峰；
- 支付回调与报名流量相互挤压；
- 后台大查询拖垮小程序主链路；
- 单副本故障导致用户无法下单或支付状态错乱；
- 异步队列积压影响通知、补偿和对账。

所以正式生产必须从第一阶段就按高可用做，而不是先单机、后重构。

## 3. 推荐生产基线

### 3.1 首个正式生产版本

建议 staging / prod 从一开始就使用：

- **云 WAF / SLB / API Gateway**
- **托管 K8s 单集群** 或等价托管容器服务
- **多可用区部署**
- **PostgreSQL 主库 + 至少 2 个只读副本**
- **Redis 高可用 / Cluster**
- **BullMQ**
- **对象存储 + CDN**

`Kafka / 云 MQ` 在首个生产版本中可以不是必选，但架构必须预留接入点。

## 4. 运行时职责与扩容策略

| 运行时 | 作用 | 扩容策略 |
| --- | --- | --- |
| `client-api` | 微信小程序 / H5 / App 主业务流量 | HPA 按 CPU + RPS + P95 延迟扩容 |
| `admin-api` | 平台后台 / 俱乐部后台 | 独立扩容，避免导出与查询拖垮 C 端 |
| `open-api` | 支付回调 / 开放接口 / 第三方同步 | 独立限流、独立告警、独立灰度 |
| `worker` | BullMQ 消费、outbox relay、通知、重试、补偿 | 按队列积压数与消费延迟扩容 |
| `scheduler` | 对账、活动状态推进、快照、周期任务 | 多副本部署，但同一任务只允许 leader 执行 |
| `ws-gateway` | 实时席位推送、通知推送 | 按连接数和广播压力独立扩容 |
| `report-job` | 导出、重报表、批处理 | 独立资源池，禁止与主 API 共享 CPU |

## 5. Kubernetes / 容器服务约束

### 5.1 副本与故障域

- 所有核心 API 至少 2 个副本，正式生产建议 3 个起步；
- 副本跨可用区分布；
- 必须启用：
  - `startupProbe`
  - `readinessProbe`
  - `livenessProbe`
  - `PodDisruptionBudget`
  - `HorizontalPodAutoscaler`

### 5.2 HPA 建议

- `client-api`：按 CPU + RPS + P95
- `worker`：按队列积压数 + 消费延迟
- `ws-gateway`：按连接数 + CPU

### 5.3 灰度与发布

- 默认滚动发布；
- 高风险功能按租户白名单或百分比灰度；
- 支付、退款、报名核心链路改动建议 canary。

## 6. 流量治理与负载均衡

### 6.1 网关层

网关必须支持：

- 路径路由
- IP / Token / Tenant 限流
- 黑白名单
- 熔断
- 灰度规则

标准路径分流：

- `/api/client/*` -> `client-api`
- `/api/admin/*` -> `admin-api`
- `/api/open/*` -> `open-api`
- `/ws/*` -> `ws-gateway`

### 6.2 应用层

- API 服务必须无状态；
- 热点活动按 `activity_id` 做动态限流；
- 支付回调入口必须与普通业务流量隔离；
- 后台导出必须走异步，不允许同步拖主库。

## 7. PostgreSQL 运维基线

### 7.1 拓扑

- 1 个主库
- 至少 2 个只读副本
- 自动备份
- 慢 SQL 监控
- 归档与冷热分层策略

### 7.2 查询策略

- 报名、支付、钱包、退款全部写主库；
- 列表、报表、审计优先走只读副本；
- 写后立即读的确认页必须走主库或强一致缓存。

### 7.3 数据变更

- 所有 schema 变更通过 Prisma Migrate 管理；
- 高风险变更必须先在 staging 回放；
- 涉及大表结构变更时要准备回滚和窗口期。

## 8. Redis 与 BullMQ 运维基线

Redis 至少逻辑上拆分为：

- `seat`: 活动席位与 Lua 原子扣减
- `rate-limit`: 限流
- `cache`: 活动详情、排期快照、热点列表
- `queue`: BullMQ 队列
- `session`: token 黑名单与短期状态

对头部客户，建议 `queue` 与 `seat/rate-limit` 至少逻辑隔离，避免队列积压影响核心链路。

BullMQ 负责：

- 订单超时关闭
- 候补递补
- 通知重试
- 异步导出
- 失败补偿

必须支持：

- 重试
- 指数退避
- 死信记录
- 人工补偿

## 9. Kafka / 云 MQ 预留策略

首个生产版本不强制全量上 MQ。  
但以下场景建议保留未来接入能力：

- 通知广播
- 数据同步
- 审计日志转储
- 异步报表
- 高吞吐事件削峰

核心事务链路仍以：

- PostgreSQL 事务
- Redis
- BullMQ
- Outbox

为主，不把强一致依赖直接建立在 MQ 上。

## 10. 可观测性

### 10.1 指标

至少覆盖：

- API QPS
- 错误率
- P95 / P99
- 热点活动报名成功率
- 支付成功率
- 支付回调重复率
- BullMQ 队列积压数
- Redis 命中率与慢命令
- PostgreSQL 主从延迟

### 10.2 日志

- 结构化 JSON
- 至少包含：
  - `trace_id`
  - `tenant_id`
  - `user_id`
  - `request_id`
  - `path`
  - `duration_ms`

### 10.3 错误与链路

- Sentry：前后端统一
- OpenTelemetry：HTTP、Redis、PostgreSQL、BullMQ 全链路追踪

## 11. 安全与合规

| 维度 | 措施 |
| --- | --- |
| 传输 | 外网 TLS，内网建议 mTLS |
| 鉴权 | JWT + RBAC + 多租户强隔离 |
| 支付 | 回调验签、状态机、幂等入账 |
| 审计 | 后台写操作、跨租户查看、退款、余额调整全部落审计日志 |
| 防爆破 | 登录失败限制、验证码、热点接口限流 |
| 数据 | 敏感字段加密或脱敏返回 |

## 12. 备份与容灾

- PostgreSQL：每日全量 + WAL / 增量备份
- Redis：托管快照或 AOF/RDB
- OSS/COS：跨区域备份
- BullMQ：队列状态可恢复，补偿任务可重放

每季度至少演练一次：

- PostgreSQL 恢复
- Redis 重建与预热
- 支付回调补偿
- 单 AZ 故障切流

## 13. 参考容量规划

| 指标 | 首个试点生产 | 稳定扩容期 |
| --- | --- | --- |
| 俱乐部数 | 10-50 | 50-200 |
| 日活会员 | 1w-5w | 5w-20w |
| 峰值 QPS | 500-1500 | 1500-5000 |
| 热点活动报名峰值 | 300-800 QPS/活动 | 800-2000 QPS/活动 |
| API 副本 | 7-18 | 18-50 |
| Worker 副本 | 2-6 | 6-20 |
