# 03 · 业务模块设计

> 本文从领域视角拆解每个核心模块的：**实体**、**关键流程**、**状态机**、**关键规则**。
> 数据库表结构详见 [04-数据库设计](./04-数据库设计.md)，接口详见 [05-后端接口设计](./05-后端接口设计.md)。

## 1. 多租户与平台管理

### 1.1 实体
- **Platform**：平台运营方（唯一）。
- **Tenant（俱乐部）**：SaaS 租户，独立运营单元。
- **TenantStaff**：俱乐部员工（含主理人、教练、前台、财务）。
- **PlatformStaff**：平台运营人员。
- **VenuePartner（场馆合作方）**：物理场馆的运营方，可与多个俱乐部合作。

### 1.2 入驻流程
```
俱乐部主理人 ─► 提交入驻申请（资质照、营业执照、收款信息）
        │
        ▼
平台运营审核（驳回 / 通过）
        │ 通过
        ▼
系统初始化租户：
  · 生成 tenant_id
  · 创建主账号 + 默认角色（主理人/前台/财务/教练）
  · 默认套餐（试用 30 天）
        │
        ▼
邮件 + 短信通知主理人 → 登录后台 → 完成基础信息引导
  · 俱乐部资料
  · 场馆 / 合作场馆
  · 收款配置
  · 默认活动模板
  · 成员导入
  · 微信群分享海报
```

### 1.3 主理人迁移工具

MVP 要优先解决「从微信群接龙迁移到系统」的阻力：

- **成员导入**：Excel / CSV 导入手机号、昵称、标签、初始余额 / 次卡备注，导入失败给出可下载错误报告。
- **活动模板**：固定时间、固定场馆、固定人数、固定价格、取消规则、分享海报样式可保存为模板。
- **周期活动**：按周复制活动，支持批量调整日期、价格、场地。
- **微信群分享**：活动详情可生成小程序分享卡片、H5 落地页、海报二维码，记录 share token。
- **运营复盘**：活动结束后展示报名人数、候补人数、爽约人数、收入、退款、分享转化。

### 1.4 套餐与计费
- **试用版**：30 天，限 1 球馆 / 100 会员。
- **基础版**：年费 X，限 3 球馆 / 5000 会员。
- **专业版**：年费 Y，不限场馆 / 不限会员 + 数据看板。
- **超大客户**：定制（独立 DB / 私有部署）。

平台抽佣模型可选：
- **平台费 + 0 佣金**：包年套餐，不抽交易额。
- **0 平台费 + 抽佣**：按交易额抽 2-5%。
- **混合**（推荐）：套餐费 + 商城类抽佣（约场不抽）。

### 1.5 SaaS 订阅与续费

> 平台对俱乐部收的钱（订阅费 / 抽佣 / 增值服务费）走独立资金链路，**绝不复用会员钱包流水**。
> 关键表见 [04 §6.7-6.10](./04-数据库设计.md)：`subscription_orders`、`commission_rules`、`settlement_orders`、`ledger_entries`。

#### 1.5.1 订阅周期与状态机

```
trial ──► active ──► expiring ──► expired
              │           │
              ▼           ▼
            suspended ─► canceled
```

- `trial`：30 天试用期，到期前 7 天通知主理人；试用结束自动降级为 `expired`，前台只读、不可发活动。
- `active`：付费订阅生效中，根据 `current_period_end` 决定有效期。
- `expiring`：到期前 14 天进入提醒状态（不影响功能），后台横幅提示续费。
- `expired`：到期未续，进入只读宽限期 7 天；7 天后停用所有功能，等待续费或彻底关闭。
- `suspended`：因合规 / 违规 / 投诉被平台暂停，主理人不可登录。
- `canceled`：俱乐部主动注销 / 主理人退订；走数据保留 90 天 → 删除流程。

#### 1.5.2 续费 / 升级 / 降级

- **续费订单**：到期前 30 天可手动续费，到期前 7 天若开启自动续费触发；订单类型 `renewal`，金额 = 当前套餐年费。
- **升级**：从基础版 → 专业版，按 **剩余天数等比换算** 补差价（按日折算）；订单类型 `upgrade`。
- **降级**：从专业版 → 基础版只允许在到期日生效（不退差价），下一周期套餐变更；订单类型 `downgrade`，金额 0，仅记录变更。
- 所有订阅订单通过支付聚合（与会员充值复用 `payment_orders`，但 `biz_type='subscription'`）。

#### 1.5.3 抽佣规则

`commission_rules` 是俱乐部维度配置，支持以下维度组合：

| 维度 | 说明 |
| --- | --- |
| `biz_type` | activity / mall / lesson / tournament，按交易类型分别配置 |
| `rate_type` | percent（百分比抽佣，如 3%）/ fixed（固定金额，如每单 0.5 元）/ none（不抽） |
| `min_per_order` | 单笔最低抽佣（避免 1 元小单抽 0.03 元） |
| `cap_per_order` | 单笔抽佣上限（保护大客户） |
| `effective_from` / `effective_to` | 生效时间，可建未来生效的规则做调价预告 |

每笔交易支付成功 → 写 `ledger_entries`：
```
D: tenant_payable    交易金额  → 应付俱乐部
C: cash              交易金额  → 平台收到的钱
D: commission_income 抽佣金额  → 平台抽佣收入
C: tenant_payable    抽佣金额  → 从应付中扣除
```

#### 1.5.4 结算

参见 §6.5 资金链路：`settlement_orders` 按俱乐部 + 周期（默认月结，可调周结）汇总：
- `gross_amount = 该俱乐部所有交易支付总额`
- `refund_amount = 期内退款总额`
- `commission_amount = 抽佣合计（来自 commission_rules）`
- `payable_amount = gross - refund - commission`
- 主理人在后台 confirm → 平台财务批量打款 → 状态变 `paid`。

#### 1.5.5 自动化任务

| 任务 | 频次 | 行为 |
| --- | --- | --- |
| 试用到期检查 | 每日 02:00 | 到期 → `expired`，发短信 + 站内信 |
| 订阅到期前提醒 | 每日 02:30 | 到期前 14/7/3/1 天分别通知主理人 |
| 自动续费扣款 | 每日 03:00 | 开启自动续费的俱乐部，到期前 7 天发起 wallet/wechat 扣款 |
| 抽佣记账 | 实时（事件驱动） | 支付成功 outbox → 写 `ledger_entries` |
| 月度结算单生成 | 每月 1 号 04:00 | 按俱乐部生成 `settlement_orders` 草稿 |
| 月度对账 | 每日 05:00 | `wallet` + `payment_orders` + `ledger_entries` 三方对账 |

## 2. 用户与认证

### 2.1 用户体系
- 平台维度全局唯一 `user_id`（UUIDv7 / Snowflake）。
- 同一手机号自动合并；微信 unionid 关联同一用户。
- 用户在不同俱乐部下持有 **MemberProfile**（会员档案，等级、标签、钱包）。

### 2.2 登录方式
| 端 | 登录 |
| --- | --- |
| 小程序 | wx.login → openid + unionid → 后端注册/登录 |
| iOS / Android App（Taro RN 输出） | 手机号 + 短信验证码（主）；微信开放平台一键登录（次） |
| 后台 | 账号密码 + 短信二次验证；可选企业 OAuth |

### 2.3 实名认证
- 核心交易类操作（充值 ≥ 1000、提现）触发实名认证（姓名 + 身份证 + 三要素核验）。
- 实名信息加密存储，仅显示掩码。

## 3. 球馆与场地

### 3.1 实体
- **Venue（球馆）**：物理场馆，归属俱乐部，可挂多个。
- **VenueOwner / VenuePartner**：场馆运营方或合作方，负责物理场馆资料、结算联系人、合作状态。
- **ClubVenueRelation**：俱乐部与球馆的合作关系，记录自营 / 合作、结算备注、可用场地范围。
- **Court（场地）**：球馆下的具体场地，编号唯一。
- **VenueOpenRule**：营业时间规则（周一-周五 / 周末 / 节假日）。
- **CourtBlock**：场地占用块（最小预订单元，默认 30min/60min）。

> 业务上不建议把「俱乐部」和「物理球馆」强绑定：多个俱乐部可能使用同一场馆，一个俱乐部也可能在多个合作场馆组织活动。平台层应维护公共场馆目录，俱乐部通过合作关系挂载场馆，避免重复录入导致排期、导航和结算混乱。

### 3.2 排期生成
- 每天凌晨 02:00 定时任务，按规则生成 **未来 14 天** 的 `court_schedule`（场地块状态：可预订/已占用/维护）。
- 主理人可手动批量调整、批量维护。
- 节假日特殊规则可临时覆盖。

### 3.3 场地占用类型
| 类型 | 说明 | 来源 |
| --- | --- | --- |
| `available` | 可预订 | 默认 |
| `held` | 报名锁定（活动绑定） | activity 创建 |
| `booked` | 已被会员包场 | booking 订单 |
| `maintenance` | 维护中 | 后台手动 |
| `closed` | 闭馆 | 节假日规则 |

## 4. 活动与约场报名

> 这是平台**最复杂、最核心**的模块。

### 4.1 活动类型
| 类型 | 说明 | 计费 |
| --- | --- | --- |
| `open_session` | 散场（俱乐部组织，多人混打） | 单人价格，N 人封顶 |
| `private_court` | 包场 | 按场地 × 时长 |
| `coach_lesson` | 教练课 | 课时单价，1v1 / 1vN |
| `tournament` | 内部比赛（二期） | 报名费 |

### 4.2 散场活动状态机
```
draft ──► published ──► registering ──► full ──► ongoing ──► finished
   │           │              │            │         │
   │           │              ▼            ▼         ▼
   └─────────► canceled ◄─ canceled ◄── canceled ── settled
```
- `draft`：草稿（编辑中）
- `published`：已发布，未到报名开始
- `registering`：报名中
- `full`：报满（仍可候补）
- `ongoing`：活动进行中（开始时间到达，结束时间未到）
- `finished`：已结束（待结算）
- `settled`：已结算（费用最终化）
- `canceled`：取消（任意阶段，需触发退款）

### 4.3 报名抢位（防超卖核心）
**关键诉求**：N 个席位多人同时点报名，绝不超卖、绝不重复扣款。

**方案：Redis 原子计数 + DB 唯一索引兜底 + 状态机 + 幂等键**

```
用户点击「报名」
   │ 携带 idempotency_key
   ▼
应用层校验：登录态、活动状态、用户是否已在报名列表
   │
   ▼
Redis Lua 脚本原子操作：
   IF activity:left > 0 THEN
       DECR activity:left
       SADD activity:participants user_id
       RETURN OK
   ELSE
       RETURN OUT_OF_STOCK
   END
   │
   ▼
（成功）扣钱包余额 / 创建支付订单（事务）
   ↓ DB 活跃报名唯一键 active_user_key 防止双报
   ↓ activity_registration 写入
   │
   ▼
（成功）写 outbox 事件 → worker / BullMQ 处理通知 / 推送 / 数据更新
```
- **失败回滚**：Redis 自动恢复（compensate 队列），最终一致。
- **候补**：`activity:left == 0` 时进入 `waiting_list`（Redis ZSET，score=入队时间）。
- **取消**：同事务内执行四件事：
  1. 把对应 `registration_participants` 行的 `status` 置为 `canceled / refunded / no_show`；
  2. **`seat_no` 置 NULL**（PostgreSQL 唯一索引同样允许多个 `NULL`，确保候补可以接管该席位号）；
  3. 把 `activity_registrations.active_user_key` 置 NULL（释放活跃报名唯一键，允许该用户重新报名）；
  4. Redis 端 `INCRBY activity:{id}:left N`、`SREM activity:{id}:participants user_id`，并触发候补递补。
- **多人代报**：`activity_registration` 表示一次报名订单；具体占用名额拆成 `registration_participants`，每个名额可独立签到、退款、标记爽约。
- **分享追踪**：报名请求可携带 `share_token`，用于归因到分享人 / 分享渠道 / 活动海报。
- **退款规则**（可配）：
  - 开赛 24h 前取消，全额退；
  - 开赛 2h 前取消，退 50%；
  - 开赛 2h 内取消，不退；
  - 由俱乐部主动取消活动，全额退 + 补偿券。

### 4.4 包场预订
- 用户选时间段 + 场地 → 预占 5min（hold）→ 支付成功 → 正式占用。
- 5min 内未支付自动释放。
- 已占用时段二次抢占走唯一索引 `(venue_id, court_id, time_slot)` UNIQUE 兜底。

> 包场对场馆排期准确性要求高，建议 M2 以后再完整开放；MVP 可先仅支持主理人创建散场活动并占用场地块。

## 5. 会员管理

### 5.1 会员档案 (MemberProfile)
- (tenant_id, user_id) 唯一。
- 字段：等级、积分、累计消费、入会日期、生日、标签、备注、来源渠道。
- 会员卡：
  - **次卡**：限定次数（10 次散场卡）。
  - **时长卡**：30 / 90 / 365 天无限报名。
  - **储值卡**：现金充值。
- 会员等级：
  - V1（新会员）/ V2 / V3 / V4 / V5；按累计消费自动升级。
  - 等级权益：折扣、专属活动、优先报名、商城额外赠品。

### 5.2 关键规则
- 一人多俱乐部：同一手机号在不同俱乐部各自有独立的会员档案与钱包。
- 黑名单：俱乐部可拉黑会员，禁止报名 + 提示。
- 标签：自定义标签（如「专业」「双打」「实习生」）便于运营筛选。

## 6. 钱包与支付

### 6.1 钱包模型
- 每个 `(tenant_id, user_id)` 一张钱包。
- 余额拆分为：
  - `cash_balance`：现金余额（充值入账，仅可消费 + 原路退回）
  - `gift_balance`：赠送余额（活动赠送，仅可消费、不可退）
- 余额展示总和，消费时按 **gift 优先 → cash 兜底** 的策略扣减。

#### 6.1.1 合规边界（金融红线）

| 能力 | 是否提供 | 原因 |
| --- | --- | --- |
| 充值（用户付钱进钱包） | ✅ | 走微信支付 / 支付宝官方通道 |
| 钱包内消费 | ✅ | 仅在该俱乐部消费 |
| 退款回钱包 | ✅ | 业务退款产生的退回，限定回到原 cash_balance |
| 退款原路回原通道（微信支付 → 微信） | ✅ | 限定与原支付方式一致 |
| **提现到银行卡 / 微信零钱** | ❌ | 预付费类钱包提现需要支付牌照（《非金融机构支付服务管理办法》），平台不持牌一律不做 |
| **跨俱乐部转账 / 余额迁移** | ❌ | 同上，会被认定为代理资金转移 |
| **赠送余额 (gift_balance) 退款 / 提现** | ❌ | 赠送资金本不属于用户实际付款，不可变现 |
| **俱乐部解散 / 跑路时余额自动提现** | ❌ | 由商务合同约定保证金 / 保险机制处理，不在系统内自动操作 |

> 后台开发者一旦看到"提现"需求，必须停下来与法务确认；任何线上 / 线下变现路径都需要额外资质评估，绝不在 v1.x 实现。

### 6.2 钱包流水与平台结算账本
会员钱包余额变化记录为 `wallet_transaction`：
```
type        | direction | amount | source_type      | source_id
recharge    | credit    | 100.00 | recharge_order   | RC2026...
recharge    | credit    |  10.00 | recharge_order   | RC2026...  (赠送)
consume     | debit     |  30.00 | activity_register| AR2026...
refund      | credit    |  30.00 | activity_register| AR2026...
```
- 每笔交易必须有对应的对端凭证（订单/退款/活动）。
- 余额 = 历史 credit 总和 - 历史 debit 总和（每日定时核对快照）。
- 平台与俱乐部之间的抽佣、结算、分账不直接复用会员钱包流水，使用 `settlement_orders` + `ledger_entries` 做经营账审计。
- Sprint 0 / M1 必须明确收款主体：平台统一收款、微信支付服务商分账、俱乐部自有商户号三选一，不在同一版本混用。

### 6.3 充值
- 充值套餐：充 100 得 110、充 500 得 600、充 1000 得 1300。
- 充值订单状态：`created → paying → paid / canceled`。
- 调用聚合支付（微信支付 / 支付宝），回调验签后入账。
- 充值送的赠送金额单独入 gift_balance。

### 6.4 退款
- 任何业务退款都创建 `refund` 记录，状态：`pending → success / failed`。
- 仅原路退还：钱包扣款 → 退回钱包；微信支付 → 退回微信。
- 部分退款 / 全额退款都支持。

## 7. 商城

### 7.1 阶段策略

商城分为两层，避免 1.0 过早进入完整电商复杂度：

- **轻商城（M3）**：次卡、时长卡、课时包、活动券包、场馆自提商品。重点是线上购买、线下核销、复购转化。
- **完整商城（二期）**：球拍、球、服饰、礼品等实物商品，补齐 SKU、库存、物流、售后、品牌入驻。

### 7.2 商品与 SKU
- `product`（商品）：基础属性、品类、品牌、上架状态。
- `product_sku`：规格组合 + 价格 + 库存。
- 商品归属：可平台自营、可俱乐部上架（以 `owner_type` 区分）。
- 轻商城商品可设置 `fulfillment_type = virtual / self_pickup / express`，MVP/M3 仅开放 `virtual` 与 `self_pickup`。

### 7.3 购物车 → 订单 → 支付 → 履约
```
购物车（Redis Hash 持久化 + 定期同步 DB）
   │
   ▼
预下单（计算优惠、运费、库存校验）
   │
   ▼
下单（创建订单、锁库存、生成 out_trade_no）
   │
   ▼
支付（钱包 / 微信支付 / 支付宝；可混合：余额抵扣 + 在线支付）
   │
   ▼
发货（自营物流接入快递鸟 / 顺丰；自提仓核销码）
   │
   ▼
确认收货 / 自动确认（10 天）
   │
   ▼
售后（仅退款 / 退货退款 / 换货）
```

### 7.4 订单状态机
```
pending_pay ─► paid ─► shipped ─► delivered ─► completed
     │           │         │           │
     ▼           ▼         ▼           ▼
   canceled   refunding  refunding  refunding
```

### 7.5 库存
- 普通预占（下单 → 锁库存 → 支付成功扣减 / 取消释放）。
- 高并发活动商品采用 Redis 预热库存 + 异步落库。

## 8. 教练与课程（二期重点，但一期预留模型）

### 8.1 实体
- `coach`：教练档案（一对一关联 user）。
- `coach_course`：课程（1v1、1v4、私教）。
- `coach_schedule`：教练空闲时段。
- `lesson_order`：课程订单（绑定到 activity 的 coach_lesson 类型）。

### 8.2 流程
- 学员选课 → 选时段 → 选场地 → 下单 → 支付 → 上课 → 教练扫码签到/确认 → 结算给教练。
- 教练分账：支付成功后冻结教练应得金额，T+1 解冻 + 结算到教练账户。

## 9. 营销

### 9.1 优惠券
- 类型：满减、折扣、立减、品类券、活动券。
- 发放方式：领取（活动）、定向（生日券、回流券）、付费购买。
- 核销：下单时选择，按规则校验门槛。
- 库存：发行总量、单人限领。

### 9.2 拼团
- 团长发起 → 邀请成团 → 满 N 人成团成功 → 全员享拼团价 → 否则原路退款。
- 适用：活动报名 + 商品。

### 9.3 邀请有礼
- 老带新：A 邀请 B → B 完成首次充值 → A、B 各得券 / 余额。
- 防作弊：实名 + 设备指纹 + IP 风控。

### 9.4 营销活动
- 后台可配置专题页（拖拽生成）→ 小程序首页位 / 活动页。

## 10. 轻赛事与积分

### 10.1 范围

轻赛事服务俱乐部留存，不做大型赛事直播：

- 周赛 / 排位赛报名。
- 后台手动分组、录入赛果。
- 积分榜、段位榜、胜负记录。
- 赛事活动可复用活动报名和支付能力。

### 10.2 实体

- **Tournament**：赛事，归属俱乐部，可绑定球馆和活动时间。
- **TournamentRegistration**：赛事报名，可复用会员 / 钱包 / 支付。
- **Match**：单场比赛，记录对阵、比分、胜方。
- **RankingPointLog**：积分流水，记录赛果带来的积分变化。

### 10.3 关键规则

- MVP 后优先支持「手动录入赛果 + 自动算积分」，暂不做复杂自动排程。
- 积分榜按俱乐部维度隔离，跨俱乐部榜单进入二期。
- 赛事取消、退款规则复用活动取消规则。

## 11. 通知与消息

### 11.1 触发场景
- 报名成功 / 候补成功 / 活动取消 / 即将开赛 30min 提醒。
- 充值成功 / 余额变动 / 优惠券到账。
- 商城订单：支付、发货、签收。

### 11.2 通道
| 通道 | 用途 |
| --- | --- |
| 微信小程序订阅消息 | 重要业务通知 |
| 微信公众号模板消息 | 长期通知（可选） |
| App 推送（极光 / 个推） | iOS / Android App（Taro RN 输出） |
| 站内信 | 全平台兜底 |
| 短信 | 关键场景（验证码、退款失败） |

### 11.3 实时席位推送
- WebSocket 长连接，按 `topic = activity:{id}` 订阅。
- 报名 / 取消触发 broadcast 增量更新（防止 N 个人都拉全量）。
- MVP 可先用短轮询降低复杂度，爆款活动、App 高频场景再开启 WebSocket。

## 12. 数据看板

### 12.1 实时看板（首页）

- 商品报表：销售排行、滞销品。

- 实时：PostgreSQL + Redis 计数器（首页关键指标）。

- 报表导出：异步任务，OSS 生成下载链接。
