# 01 · 项目概述与需求分析

## 1. 业务背景

国内羽毛球运动渗透率高，城市俱乐部数量庞大，但大量俱乐部仍在使用 **微信群 + 接龙 + Excel** 的方式运营，存在以下痛点：

- 报名 & 到场签到流程混乱，扣费靠人工。
- 会员卡、储值卡、次卡缺乏统一管理，对账困难。
- 球馆排期与场地占用情况不透明，超额报名/抢场争议多。
- 无沉淀的会员资产和数据分析，难以做精细化运营。
- 相关周边（球拍、球线、装备）销售分散在私域，没有形成闭环。

对标产品 **闪动**：重点学习其「俱乐部供给活动 → 球友发现并报名 → 社群传播与复购」的增长飞轮，而不是在 1.0 直接复刻完整社区、商城、赛事平台。本项目目标是先构建一套 **可对外销售的羽毛球俱乐部约场 SaaS**，让俱乐部主理人能从「微信群 + 接龙 + Excel」低成本迁移到系统，并沉淀会员与交易数据。

## 2. 平台定位

- **多租户 SaaS**：每个俱乐部即一个租户（tenant），独立运营、独立资金账户。
- **B + C 双端**：B 端给俱乐部 / 平台运营人员使用（Web 后台），C 端给会员 / 球友使用（小程序 + H5 + iOS + Android，一套 Taro 代码）。
- **轻量为先**：起步阶段聚焦「活动报名 + 会员/卡券 + 钱包/支付 + 分享裂变 + 基础财务」五件事；完整实物商城、复杂社区、直播赛事等放到二期。

## 3. 用户角色与权限

| 角色 | 端 | 主要权限 |
| --- | --- | --- |
| 平台超级管理员 | 后台 | 俱乐部入驻审核、平台级配置、平台佣金、风控 |
| 平台运营 | 后台 | 内容审核、客服、营销活动、Banner |
| 俱乐部主理人 | 后台 | 俱乐部信息、球馆/场地、活动、订单、财务、员工 |
| 俱乐部员工（前台/教练助理） | 后台 / 小程序 | 排课、签到、核销、会员服务 |
| 场馆合作方 / 场馆运营 | 后台（可选） | 场地排期、维护时段、价格、场馆结算 |
| 教练 | C 端四端 | 课程列表、学员管理、签到 |
| 会员 / 普通球友 | C 端四端 | 浏览俱乐部、报名活动、约场、充值、商城下单、查看订单 |
| 游客 | C 端四端 | 浏览公开信息、注册登录 |

> 角色权限在后台采用 **RBAC**（角色-权限-用户三段式）模型；俱乐部内角色支持自定义菜单授权。

## 4. 核心业务场景

### 4.1 俱乐部入驻
1. 俱乐部主理人小程序/PC 提交资质 → 平台审核 → 创建租户、初始化默认套餐 → 主理人登录后台。
2. 进入后台后引导完成：俱乐部基础信息 → 球馆 & 场地 → 收款配置 → 活动模板 → 微信群分享海报 → 邀请 / 导入会员。
3. 对从微信群迁移的俱乐部，提供 Excel 成员导入、历史活动模板复用、默认取消规则、默认报名价格，降低主理人首次配置成本。

### 4.2 球馆 & 场地管理
- 一个俱乐部可以挂载多个球馆（合作的或自营的）。
- 一个物理球馆也可能被多个俱乐部使用，平台需要维护「俱乐部 - 场馆」合作关系，避免同一场馆被重复录入导致排期和结算混乱。
- 一个球馆下有 N 块场地，支持设置场地编号、类型（标准 / 训练）、是否启用。
- 营业时段、节假日特殊排期、场地维护时间。

### 4.3 约场报名（核心）
两种典型形态，按优先级分阶段建设：
- **散场约场（MVP 核心）**：俱乐部发布 N 时段的散场活动，会员报名抢席位（如 16 人封顶），到时间到场打球；超时未到自动扣费 / 提前取消可退。
- **包场预订（M2 起增强）**：会员选某个时间段的某块场地直接预订（按小时计费），需要更强的场馆排期准确性和 5min 预占机制。

需要支持：
- 抢位防超卖：Redis Lua 原子计数 + DB 唯一索引 / 事务兜底。
- 报名后的取消规则：N 小时前可退、N 小时内不可退。
- 候补 / 排队：满员后进入候补，前面有人取消自动递补并通知。
- 多人代报：会员可一次报多个名额（带球友）。
- 实时席位变化推送（WebSocket）。
- 分享裂变：活动详情可生成微信群海报 / H5 落地页，支持通过 share token 追踪来源、邀请人和报名转化。

### 4.4 会员管理
- 普通会员（免费）/ 付费会员（月卡/季卡/年卡）。
- 会员等级（消费金额 + 活跃度自动升级）。
- 会员标签（实习生 / VIP / 黑名单）。
- 入会、续费、转赠、迁移俱乐部。

### 4.5 钱包 & 充值
- 每个会员对每个俱乐部独立钱包（充值仅在该俱乐部使用）。
- 平台钱包（可选，用于跨俱乐部商城消费）。
- 充值送：充 100 送 10、充 500 送 80。
- 储值类型：现金余额 / 赠送余额（**仅限在该俱乐部消费、不可提现、不可跨俱乐部转移**）。
  - 退款只能原路退回（钱包扣款 → 退回钱包；微信支付 → 退回微信），**不提供提现到银行卡 / 微信零钱的功能**。这是预付费类钱包的金融合规红线，避免触碰支付牌照。
  - 俱乐部经营异常（跑路 / 暂停服务）时的会员余额处置，由平台与俱乐部合同约定（保证金 / 保险），不由系统自动提现。
- 资金流水可审计、月度对账。

### 4.6 实名认证

- 触发场景（默认阈值，按地区监管在配置中心可调）：
  - **单笔充值 ≥ ¥200** 或 **当日累计充值 ≥ ¥500**；
  - 任何提现 / 退款到原通道金额 > ¥1000；
  - 平台运营标记为高风险用户。
- 阈值通过 `system_configs.realname_threshold_*` 配置；不同省份 / 城市监管要求不同时，可按 `tenant.region` 覆盖默认值。
- 实名信息加密存储（AES-GCM + KMS），仅显示掩码；通过三要素 / 四要素核验接口（身份证 + 姓名 + 手机号 + 银行卡）。

### 4.7 商城
- **M3 轻商城**：优先支持次卡、时长卡、课时包、活动券包、场馆自提商品，围绕俱乐部复购与线下核销。
- **二期完整商城**：再扩展球拍、球、服饰等实物商品，补齐 SKU、物流、售后、库存流水。
- 与会员钱包 / 优惠券打通，支持余额、赠送余额、券包抵扣。

### 4.8 教练与课程（二期重点）
- 教练资料、课时单价、空闲时段。
- 一对一 / 一对多课程报名。
- 课时卡核销。

### 4.9 营销
- 优惠券（满减、折扣、立减、限定品类/活动）。
- 拼团、邀请有礼、分享得券。
- 活动 Banner、首页推荐位。

### 4.10 轻赛事与积分
- M2 先做轻赛事：俱乐部周赛 / 排位赛报名、分组、赛果录入、积分榜、段位榜。
- 不做直播、复杂编排和跨城大型赛事，先服务俱乐部留存和会员复购。

### 4.11 数据看板
- 经营总览：交易额、活跃会员、约场场次、商品销售。
- 财务对账：充值、消费、退款、平台抽成。
- 报表导出（CSV / Excel）。

## 5. 非功能性需求

| 维度 | 目标 |
| --- | --- |
| 可用性 | 99.9%；核心约场接口 99.95%，主备切换 < 1min |
| 性能 | 单租户 1w 会员，约场报名抢位 P99 < 300ms；商城列表 < 200ms |
| 并发 | 抢爆款活动场景：单活动峰值 500 QPS，全局 5k QPS |
| 数据安全 | 用户密码 bcrypt；手机号脱敏存储；支付回调签名验签 |
| 多租户隔离 | 租户级数据严禁泄漏，所有查询强制 tenant_id 过滤 |
| 合规 | 个保法 / 等保 2.0 三级；支付宝 / 微信支付资质对接 |
| 国际化 | 一期中文为主，预留 i18n 接入点 |

## 6. 产品边界与非目标

**一期不做**：
- 直播 / 短视频。
- 线下打卡机硬件对接（仅预留二维码核销）。
- 基于 LBS 的社交相亲式约球（仅做基础位置筛选）。
- 跨俱乐部的钱包互通（资金合规链路复杂，二期评估）。
- 完整实物电商、物流、售后体系。
- 独立 RN 双工程；C 端统一采用 Taro 单仓四端方案。
- 大型微服务拆分与重型 K8s 基础设施，MVP 采用单体 + DDD 分包起步。

**二期评估**：
- 完整商城（实物 SKU / 物流 / 售后 / 品牌入驻）。
- 比赛报名 + 赛事直播。
- 球友社区 / 动态 / 战队。
- 教练 KOL 体系。
- 智能推荐（活动、商品、教练）。

## 7. 关键风险与对策

| 风险 | 对策 |
| --- | --- |
| 抢位超卖 / 重复扣款 | Redis Lua 原子计数 + 数据库唯一索引 + 幂等键 |
| 资金对账差错 | 钱包流水 + 支付 / 退款单 + 结算账本分层，每日对账 + 异常告警 |
| 多租户数据串号 | 强制 tenant_id 中间件、ORM 全局 Scope、CR 检查 |
| 业务复杂度膨胀 | 单体起步 + DDD 分包，明确各模块边界，按 GMV 拆服务 |
| 合规（支付、个保） | 一开始就接微信/支付宝官方支付，敏感字段脱敏存储 |
| 主理人迁移成本高 | 活动模板、成员导入、微信群分享、默认配置、客户成功陪跑 |
| 资金合规路径不清 | 一期明确统一收款 / 服务商分账 / 俱乐部自有商户号三选一，不混用 |
| 四端一致性风险 | Taro 单仓、PlatformAdapter、四端 CI 构建与核心路径自动化测试 |
