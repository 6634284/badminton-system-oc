# 04 · 数据库设计

> 数据库：**PostgreSQL 16**  
> 多租户策略：**共享库 + 行级 `tenant_id` 隔离**（平台级表除外）  
> 主键：业务表统一使用 **应用层生成的 `BIGINT` 雪花 ID**，同时维护业务编号 `xxx_no`  
> 通用时间字段：统一使用 **`TIMESTAMPTZ(3)`** 语义，由数据库按 UTC 存储  
> ORM：**Prisma 为主，热点链路允许 Raw SQL**

> 本文是数据库设计与后端实现的共同基线。  
> 其中涉及高并发报名、支付、退款、钱包、对账的表结构和索引视为硬约束，后续若要调整，必须同时更新接口设计与实施细则。

## 0. 通用规范（强制约束）

### 0.1 类型规范

- **主键 / 外键**：统一使用 `BIGINT`
  - PostgreSQL 没有 `UNSIGNED BIGINT`，但应用层生成的雪花 ID 必须始终为正数。
- **金额字段**：统一使用 `NUMERIC(12,2)` 起步
  - 禁止 `REAL / DOUBLE PRECISION`
- **时间字段**：统一使用 `TIMESTAMPTZ`
  - 不使用 `TIMESTAMP WITHOUT TIME ZONE`
  - `play_date` 这类纯业务日历字段允许使用 `DATE`
  - `updated_at` 由应用层统一更新；首版不依赖数据库触发器自动维护
- **布尔字段**：统一使用 `BOOLEAN`
- **配置 / 元数据字段**：统一使用 `JSONB`
  - 仅用于配置和快照
  - 高频过滤字段必须冗余为实体列或加表达式索引
- **敏感加密字段**：统一使用 `BYTEA`
  - 例如实名信息、证件号
- **文本字段**：统一使用 `TEXT` / `VARCHAR(n)`

### 0.2 状态字段规范

- 业务状态优先使用 **字符串枚举**
- 不使用数值 enum
- 数据库层可用 `CHECK` 约束保证状态合法
- Prisma 层可再映射为 `enum`

推荐风格：

- `draft`
- `published`
- `registering`
- `paid`
- `canceled`

### 0.3 JSONB 使用规范

- `JSONB` 适用于：
  - 配置
  - 规则
  - 快照
  - 第三方返回原文脱敏存档
- 禁止把核心过滤条件只放进 `JSONB`
- 若后续查询频繁，应：
  - 冗余独立列
  - 或建立 `expression index`
  - 或建立 GIN 索引

### 0.4 外键策略

- **高写入业务表默认不加 FK**
  - 原因：热路径性能、演进灵活性、异步补偿、跨库与冷热分离需求
- **小型字典表、桥接表可以加 FK**
  - 如 `role_permissions`
- 所有逻辑上的“外键字段”必须有索引，即使没有物理 FK

### 0.5 ID 与编号规约

| 类型 | 用途 | 类型 | 生成方 |
| --- | --- | --- | --- |
| 主键 ID | 内部主键、跨表关联 | `BIGINT` | 应用层 Snowflake 生成器 |
| 业务编号 `xxx_no` | 客服查单、运营展示、外部引用 | `VARCHAR(64)` | 应用层生成 |
| 三方支付单号 `out_trade_no` | 微信 / 支付宝 / 通道 | `VARCHAR(64)` | 应用层生成 |

### 0.6 命名规范

- 表名：`snake_case` 复数
- 字段：`snake_case`
- 外键字段：`xxx_id`
- 业务编号：`xxx_no`
- 时间：`xxx_at`
- 状态：`status`
- 类型：`type`
- 金额：`xxx_amount`
- 索引：
  - `uk_xxx`
  - `idx_xxx`
  - `gin_xxx`
  - `pt_xxx`（partition template 或分区辅助命名）

### 0.7 软删除与归档

- 软删除字段统一为 `deleted_at TIMESTAMPTZ`
- 流水类表：
  - `wallet_transactions`
  - `ledger_entries`
  - `audit_logs`
  - `outbox_events`
  只追加，不更新、不删除
- 3 个月以上订单 / 流水可归档到冷库或 OLAP

### 0.8 Prisma 映射约定

- `BIGINT` -> Prisma `BigInt`
- `NUMERIC(12,2)` -> Prisma `Decimal`
- `JSONB` -> Prisma `Json`
- `BYTEA` -> Prisma `Bytes`
- `TIMESTAMPTZ` -> Prisma `DateTime`
- 表名使用 `@@map("table_name")`
- 字段名统一驼峰映射到下划线列名

### 0.9 多租户硬约束

- 所有租户表必须带 `tenant_id`
- 高频索引首列优先是 `tenant_id`
- 所有业务查询默认带 `tenant_id`
- 平台越权查询必须走显式系统作用域并记录审计日志

## 3. 用户与会员

### 3.1 `users`

```sql
CREATE TABLE users (
  id BIGINT PRIMARY KEY,
  phone VARCHAR(20),
  phone_hash CHAR(64),
  email VARCHAR(128),
  nickname VARCHAR(64),
  avatar_url TEXT,
  gender SMALLINT DEFAULT 0,
  birthday DATE,
  union_id VARCHAR(64) UNIQUE,
  real_name BYTEA,
  id_card BYTEA,
  real_verified BOOLEAN NOT NULL DEFAULT false,
  password_hash TEXT,
  status VARCHAR(16) NOT NULL DEFAULT 'active'
    CHECK (status IN ('active', 'frozen', 'deleted')),
  last_login_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  deleted_at TIMESTAMPTZ
);

CREATE UNIQUE INDEX uk_users_phone_hash
  ON users (phone_hash)
  WHERE phone_hash IS NOT NULL AND deleted_at IS NULL;
```

说明：

- `phone` 明文存储，依赖数据库加密、网络 TLS 与返回脱敏；
- `phone_hash` 作为查询与唯一约束字段；
- `real_name` / `id_card` 使用应用层 AES-GCM + KMS，加密后以 `BYTEA` 存储。

## 12. 索引与性能策略

### 12.1 通用规则

- 租户表高频索引首列优先 `tenant_id`
- 大字段不进组合索引
- `deleted_at` 不单独建索引，统一配合部分索引使用
- 高频 JSONB 查询必须评估表达式索引或 GIN

### 12.2 热点防护

- `activities.join_count`、`waitlist_count` 高频更新：优先 Redis 计数 + 定时回刷
- `wallets.cash_balance`：条件更新 + 事务流水
- 支付回调：落库快进快出，通知和推送后移
- 活动详情：走缓存快照 + version

### 12.3 分区与演进

首个生产版本不建议过早分库，但要预留：

#### 12.3.1 月度分区

以下表优先考虑按时间做原生 PostgreSQL 分区：

- `wallet_transactions`
- `mall_orders`
- `activity_registrations`
- `audit_logs`
- `outbox_events`

推荐策略：

- `PARTITION BY RANGE (created_at)`
- 以月为单位建分区
- 归档和冷数据清理更容易

#### 12.3.2 头部客户隔离

对超大客户或私有化客户，优先方案是：

- 独立 PostgreSQL 集群
- 独立 Redis
- 独立部署环境

而不是在一个共享库里做复杂的在线分片路由。

### 12.4 备份与归档

- PostgreSQL 主从 + 每日全量 + WAL 增量备份
- 流水类只追加表，季度归档至 ClickHouse / Doris 用于报表
- 归档后业务库保留热数据和必要摘要
