# 07 · 跨端前端方案（一码四端：小程序 + H5 + iOS + Android）

> 工程：**`client-taro/` 单一仓库**，一套 React 代码同时编译产出：
> - 微信小程序、支付宝小程序、字节小程序（按需开启）
> - H5（活动分享落地页）
> - **iOS / Android 原生 App（基于 Taro 的 React Native 输出）**
>
> 核心原则：**业务逻辑 100% 共享，UI 组件优先用 Taro 跨端原语，平台差异通过条件编译 + 适配器抹平**。
> Hermes / Fabric 启用、OTA 灰度脚本、四端回归清单详见 [10-技术实施细则](./10-技术实施细则.md)。

## 1. 路线决策

### 1.1 为什么选 Taro 一码四端

Taro 4.x 同时支持小程序、H5、React Native 三类输出，其中 React Native 输出承载 iOS 与 Android 两端，最终形成「小程序 + H5 + iOS + Android」四端同源。底层把 React 渲染管线分别对齐到：
- 小程序的双线程模型（逻辑层 + 渲染层）；
- 浏览器 DOM；
- React Native 的 Yoga 布局 + 原生组件。

> 选择 Taro 而非 Taro + 独立 RN 双工程的原因：
> - 业务节奏：俱乐部约场、商城业务，UI 复杂度集中在「列表 + 表单 + 长流程」，没有重度 3D / 视频 / 复杂手势的强需求，RN 端不需要完全自定义 UI。
> - 维护成本：双工程意味着两份页面、两份导航、两份调用，长期会出现端不一致 bug；一码四端把人力集中在业务而不是端胶水。
> - 团队规模：6 人前端团队（含 1 RN 兼职）做不动两套 UI 的同等质量。

### 1.2 取舍与边界

| 取 | 舍 |
| --- | --- |
| 单仓单分支，统一发版节奏 | 极端原生交互（如 IM、复杂相机滤镜）不便实现 |
| 业务页面、表单、列表、表格 100% 共享 | 部分 RN 端高性能列表需要走平台特定实现 |
| 状态、API、模型、校验、工具 100% 共享 | 小程序与 RN 各自有独立平台限制（包大小 / 原生权限），需要小心设计 |

> 真要做高度原生定制（如比赛直播页 / 视频会议）时，**Taro 支持嵌入原生页面**：原生页用 RN 单独写，挂到 Taro 路由中（详见 §8）。这条退路保证「以一码四端为默认，但不被锁死」。

## 4. 平台条件编译

Taro 在编译期注入 `process.env.TARO_ENV ∈ {weapp, alipay, swan, tt, h5, rn}`，配合多种条件编译手段：

### 4.1 文件级（推荐）
> 命名约定：`xxx.weapp.ts` / `xxx.h5.ts` / `xxx.rn.ts`，未匹配端 fallback 到 `xxx.ts`。

```
platform/
  adapter.ts              # 默认导出（可 throw 提醒未实现）
  adapter.weapp.ts
  adapter.alipay.ts
  adapter.h5.ts
  adapter.rn.ts
```
编译时 Taro 自动按端选择对应文件，业务代码只 import `from '@/platform/adapter'`。

### 4.2 代码级
```ts
if (process.env.TARO_ENV === 'rn') {
  // RN 专属
}
```
> 仅用于零碎差异；大块差异走文件级。

### 4.3 组件级
对小部件的差异（如分享按钮在小程序是 `<Button open-type="share">`，RN 端是自定义弹层），抽出 `<ShareButton.weapp.tsx>` / `<ShareButton.rn.tsx>` 同名 sibling，由打包器自动选择。

### 4.4 样式级
```ts
// styles/stylesheet.ts
import { Platform, StyleSheet as RNStyleSheet } from 'react-native';

export function createStyle<T extends Record<string, any>>(rn: T, web?: Record<string, string>) {
  if (process.env.TARO_ENV === 'rn') return RNStyleSheet.create(rn) as T;
  return web as any; // 小程序/H5 走 className（推荐用 CSS Module / scss）
}
```

## 5. UI 组件策略

> 核心难点：小程序 + H5 + iOS / Android 公用一套组件库不存在「能直接拿来即用且覆盖全场景」的成熟方案，因此采用「**Taro 跨端原语 + 自建业务组件 + 端特化兜底**」。

### 5.1 三层组件分层

```
[L3] 业务组件   ActivityCard / ProductCard / WalletHeader …  ← 共享，仅写一次
[L2] 原子组件   Button / Input / Toast / Modal / Drawer …    ← 共享，端内用 native styling
[L1] Taro 原语  View / Text / Image / ScrollView / Swiper …  ← 框架自带，自动跨端
```

- **L1 原语**：`@tarojs/components` 自带映射；Taro 编译到 RN 时会替换为 RN 原生组件，编译到小程序时是小程序组件。
- **L2 原子**：自己写一份基于 L1 + 平台分支的实现；样式用 `createStyle()`。
- **L3 业务**：100% 跨端，禁止使用 DOM API、禁止使用小程序 / RN 私有 API。

### 5.2 关于 NutUI / taro-ui

| 库 | 小程序 | H5 | RN | 结论 |
| --- | --- | --- | --- | --- |
| NutUI-React-Taro | ✅ | ✅ | ❌（多组件依赖 DOM） | 仅 H5 + 小程序使用，RN 不引入 |
| taro-ui | ✅ | ✅ | 部分 | 同上 |
| 自建原子库（基于 Taro 原语） | ✅ | ✅ | ✅ | **核心选择**：薄、可控 |

策略：
- 小程序 + H5 端 **可以叠加使用 NutUI** 的复杂组件（如 Calendar、Tabbar、SignaturePad）以加速开发；
- 但凡需要在 RN 端使用的组件 → 必须使用自建原子组件（基于 `View/Text` + `createStyle()`）；
- 业务页直接 import `@/components/primitives/Button`，原子组件内部决定差异；
- 涉及高级交互的小程序专属组件（如 `<live-player>`、`<canvas type="webgl">`）封装为 `XxxOnlyMiniapp.weapp.tsx`，RN 端用 `null` 或替代实现。

### 5.3 设计 Token 与样式
- `styles/tokens.ts` 是 **跨端唯一来源**（颜色、字号、间距、圆角、阴影、动效时长）。
- 小程序 + H5：通过 SCSS 变量生成 `$color-primary` 等。
- RN：直接 `import { tokens }` 用对象。
- 暗色模式：tokens 提供 light / dark 两套，由 `useColorScheme()` 切换；小程序通过 page meta 配置。

## 6. 路由抽象

小程序的路由是 **Page 注册式**（声明在 `app.config.ts`），RN 用 `react-navigation` 是 **Stack/Tab 容器式**。需要抽象。

### 6.1 单一路由表
```ts
// router/routes.ts
export const routes = {
  Home:           { path: '/pages/home/index',           component: HomePage,           tab: true },
  TenantDetail:   { path: '/pages/tenant/detail/index',  component: TenantDetailPage },
  ActivityList:   { path: '/pages/activity/list/index',  component: ActivityListPage,   tab: true },
  ActivityDetail: { path: '/pages/activity/detail/index',component: ActivityDetailPage },
  Booking:        { path: '/pages/booking/index',        component: BookingPage },
  Wallet:         { path: '/pages/wallet/index',         component: WalletPage,         tab: true },
  Recharge:       { path: '/pages/recharge/index',       component: RechargePage },
  Mall:           { path: '/pages/mall/index',           component: MallPage,           tab: true },
  ProductDetail:  { path: '/pages/mall/product/index',   component: ProductDetailPage },
  My:             { path: '/pages/my/index',             component: MyPage,             tab: true },
  Login:          { path: '/pages/login/index',          component: LoginPage },
} as const;

export type RouteName = keyof typeof routes;
```

### 6.2 小程序：脚本生成 `app.config.ts`
```ts
// scripts/gen-app-config.ts
import { routes } from '../src/router/routes';
const pages = Object.values(routes).map(r => r.path.replace(/^\//, ''));
const tabBar = { list: Object.entries(routes)
  .filter(([_, r]) => r.tab)
  .map(([k, r]) => ({ pagePath: r.path.replace(/^\//, ''), text: tabText[k] })) };
fs.writeFileSync('src/app.config.ts',
  `export default ${JSON.stringify({ pages, tabBar }, null, 2)}`);
```
> 在 `npm run prebuild` 钩子中跑，避免手工同步两份。

### 6.3 RN：用同一份路由表生成 `react-navigation`
```ts
// router/nav.rn.ts
import { createNativeStackNavigator, createBottomTabNavigator } from '@react-navigation/...';
import { routes } from './routes';

const Stack = createNativeStackNavigator();
const Tab = createBottomTabNavigator();

const TabsNav = () => (
  <Tab.Navigator>
    {Object.entries(routes).filter(([_, r]) => r.tab).map(([name, r]) =>
      <Tab.Screen key={name} name={name} component={r.component} />)}
  </Tab.Navigator>
);

export const RootNav = () => (
  <Stack.Navigator screenOptions={{ headerShown: false }}>
    <Stack.Screen name="Tabs" component={TabsNav} />
    {Object.entries(routes).filter(([_, r]) => !r.tab).map(([name, r]) =>
      <Stack.Screen key={name} name={name} component={r.component} />)}
  </Stack.Navigator>
);
```

### 6.4 跨端导航 API
业务代码只用 PlatformAdapter 提供的方法：
```ts
import { platform } from '@/platform/adapter';

platform.navigate.push('ActivityDetail', { id: 123 });
platform.navigate.back();
platform.navigate.redirect('Home');
```
- 小程序内部调 `Taro.navigateTo({ url: route.path + qs(params) })`。
- RN 内部调 `navigationRef.navigate(name, params)`。

### 6.5 深链 / 分享
- 小程序：分享路径直接是 `route.path?id=123`。
- RN：`react-navigation` linking 配置，按 `routes` 自动生成 prefix `app://` / `https://m.example.com/...`。
- H5：路由模式 hash，路径与小程序一致；扫码落地后通过 UA 判断引导跳小程序或 App。

## 7. 端能力适配器（PlatformAdapter）

业务层只依赖一份类型，端内各自实现。

```ts
// platform/types.ts
export interface PlatformAdapter {
  kind: 'weapp' | 'alipay' | 'h5' | 'rn';
  storage: KVStorage;
  http:   HttpClient;
  navigate: Navigator;
  pay: { wechat: (p: WeChatPayParams) => Promise<void>;
         alipay: (p: AlipayParams)   => Promise<void> };
  share: ShareApi;
  scan(): Promise<string>;
  ws(url: string, token: string): WSConn;
  upload(file: PickedFile, target: UploadTarget): Promise<UploadResult>;
  pickImage(opt?: PickOpts): Promise<PickedFile[]>;
  location(): Promise<Coordinates>;
  notify: { subscribe(templateIds: string[]): Promise<void> };
  device: { platform: 'ios'|'android'|'mp'|'web'; version: string };
}
```

### 7.1 小程序实现（节选）
```ts
// platform/adapter.weapp.ts
import Taro from '@tarojs/taro';
const adapter: PlatformAdapter = {
  kind: 'weapp',
  storage: {
    get: async (k) => Taro.getStorageSync(k) ?? null,
    set: async (k, v) => Taro.setStorageSync(k, v),
    remove: async (k) => Taro.removeStorageSync(k),
  },
  http: { request: (opts) => /* Taro.request */ },
  navigate: { /* Taro.navigateTo / navigateBack */ },
  pay: { wechat: (p) => new Promise((res, rej) =>
    Taro.requestPayment({ ...p, success: () => res(), fail: rej })) },
  share: { /* showShareMenu */ },
  scan: () => Taro.scanCode().then(r => r.result),
  ws: (url, token) => createTaroSocket(url, token),
  upload: (file, t) => Taro.uploadFile({ ... }),
  pickImage: (o) => Taro.chooseImage(o),
  location: () => Taro.getLocation({ type: 'gcj02' }),
  notify: { subscribe: (ids) => Taro.requestSubscribeMessage({ tmplIds: ids }) },
  device: { platform: 'mp', version: Taro.getSystemInfoSync().version },
};
export default adapter;
```

### 7.2 RN 实现（节选）
```ts
// platform/adapter.rn.ts
import AsyncStorage from '@react-native-async-storage/async-storage';
import { MMKV } from 'react-native-mmkv';
import * as WeChat from 'react-native-wechat-lib';
import { Linking, PermissionsAndroid, Platform } from 'react-native';
import { navigationRef } from '@/router/nav.rn';
import axios from 'axios';

const storage = new MMKV();

const adapter: PlatformAdapter = {
  kind: 'rn',
  storage: {
    get: async (k) => { const v = storage.getString(k); return v ? JSON.parse(v) : null; },
    set: async (k, v) => storage.set(k, JSON.stringify(v)),
    remove: async (k) => storage.delete(k),
  },
  http: { request: (opts) => axiosClient.request(opts) },
  navigate: {
    push: (name, params) => navigationRef.current?.navigate(name as any, params),
    back: () => navigationRef.current?.goBack(),

  },
  pay: { wechat: (p) => WeChat.pay(p), alipay: (p) => Alipay.alipay(p.orderInfo) },
  share: { /* WeChat.shareToSession */ },
  scan: openCameraScan,        // react-native-vision-camera
  ws: createRNSocket,
  upload: rnUpload,
  pickImage: rnPickImage,      // react-native-image-picker
  location: rnGetLocation,
  notify: { subscribe: async () => { /* RN 端推送由通道接管，noop */ } },

```

### 7.3 H5 实现（节选）
H5 走浏览器 API：`localStorage`、`fetch`、`window.location`、`navigator.geolocation` 等；钱包支付 H5 端只在公众号内部跳起 JSAPI 支付，外部浏览器引导跳转小程序 / App。

## 8. 原生能力扩展

一码四端不等于「不能写原生代码」。下面三类策略，按需选用：

### 8.1 RN 既有库直接接入
绝大多数原生能力都有现成 RN 库（推送、扫码、人脸识别、相机、视频）。在 `client-taro/package.json` 增加 RN 依赖，仅在 `platform/adapter.rn.ts` 与 `native/` 中使用，不污染共享层。

| 能力 | RN 库 |
| --- | --- |
| 微信支付 / 分享 / 登录 | `react-native-wechat-lib` |
| 支付宝 | `react-native-alipay` |
| 推送 | `react-native-jpush` 或 `@react-native-firebase/messaging` |
| 相机 / 扫码 | `react-native-vision-camera` |
| 高性能存储 | `react-native-mmkv` |
| 长列表 | `@shopify/flash-list` |
| 地图 | `react-native-amap3d` |
| 热更新 | `react-native-code-push` |

### 8.2 自定义原生模块
团队定制能力（如自研 IM、第三方 SDK 没有官方 RN 包），写 Native Module：
```
ios/CustomModules/*.swift
android/app/src/main/java/.../CustomModules/*.kt
```
对 JS 暴露统一接口，业务通过 `platform.xxx` 间接调用。

### 8.3 嵌入完整原生页
特别复杂的页面（如比赛直播 / 视频会议）走「以原生为主，Taro 路由跳转」的方式：
- iOS / Android 写一段原生 ViewController / Activity；
- RN 路由用 `<Stack.Screen>` 包一层 `requireNativeComponent`；
- 业务侧 `platform.navigate.push('LivePage', { id })` 即可，调用方无感知。
- 小程序端无对应能力时返回「该功能仅在 App 内可用」提示。

## 9. 业务逻辑共享（示例）

```ts
import { v4 as uuid } from 'uuid';
import { http } from '@/services/http';

export function useRegisterActivity() {
  const qc = useQueryClient();
  return useMutation({
    mutationFn: (req: RegisterActivityReq) =>
      http<RegisterActivityResp>({
        url: `/activities/${req.activityId}/registrations`,
        method: 'POST',
        data: { extra_count: req.extraCount, pay_method: req.payMethod, coupon_code: req.couponCode },
        headers: { 'Idempotency-Key': uuid() },
      }),
    onSuccess: (_, vars) => {
      qc.invalidateQueries({ queryKey: ['activity', vars.activityId] });
      qc.invalidateQueries({ queryKey: ['wallet'] });
    },
  });
}
```

### 9.2 活动详情页（一份代码四端跑）
```tsx
// pages/activity/detail/index.tsx
import { View, Text, ScrollView } from '@tarojs/components';
import { useRoute } from '@/router';
import { useActivityDetail, useActivitySeats, useRegisterActivity } from '@/hooks/activity';
import { ActivityHeader, SeatGrid, FixedBar } from '@/components/activity';
import { platform } from '@/platform/adapter';

export default function ActivityDetailPage() {
  const { params } = useRoute<{ id: string }>();
  const id = Number(params.id);
  const { data: act, isLoading } = useActivityDetail(id);
  const seats = useActivitySeats(id);                  // WebSocket 增量推送
  const register = useRegisterActivity();

  if (isLoading || !act) return <LoadingView />;

  const onRegister = async () => {
    if (act.payMethodHint === 'wechat')
      await platform.notify.subscribe(['xxx_template_id']); // 仅小程序生效，RN 内 noop
    register.mutate({ activityId: id, extraCount: 0, payMethod: 'wallet' });
  };

  return (
    <View className="page">
      <ScrollView scrollY className="scroll">
        <ActivityHeader activity={act} />
        <SeatGrid seats={seats} />
      </ScrollView>
      <FixedBar
        price={act.price}
        loading={register.isPending}
        disabled={act.status !== 'registering'}
        onClick={onRegister}
      />
    </View>
  );
}
```
> 这段代码：在小程序里编译为 `wxml`，在 H5 编译为 div + flex，在 RN 编译为 `View/ScrollView/Text`。业务代码完全无感。

## 10. 工程化与质量

### 10.1 包管理
单仓 `pnpm`，无 workspace。多端依赖在同一个 `package.json`：
- 通用依赖（react、taro、zustand、tanstack-query、dayjs、zod）必装；
- RN 特有（react-native、metro、@react-navigation/...、mmkv、wechat-lib）按 peer 依赖；
- 小程序特有（NutUI）通过 dynamic import + 平台分支隔离，不进 RN bundle。

### 10.2 编译产物隔离
`metro.config.js` 配置 RN 端 resolver 跳过 NutUI 等 DOM-only 库，避免误打入：
```js
module.exports = {
  resolver: {
    blockList: [/@nutui\/nutui-react-taro/], // RN 不打这些
  },
};
```
反过来，RN 专属库（如 `react-native-vision-camera`）通过 Taro 的 platform branch 编译时只在 RN 命中：
```ts
// adapter.rn.ts 才 import；其他端不会触达
import { Camera } from 'react-native-vision-camera';
```

### 10.3 类型与生成
- TypeScript strict mode；
- API 类型由 NestJS Swagger 导出的 OpenAPI，通过 `swagger-typescript-api` 生成到 `src/services/api/generated/`；
- 路由名 + tab 配置脚本生成 `app.config.ts`，确保单一来源。

### 10.4 测试
- 单元测试：Vitest + Testing Library，跑 hooks / models / utils。
- 组件测试：原子组件按端 mock；业务组件至少跑一遍小程序端快照。
- E2E：
  - 小程序：`miniprogram-automator`；
  - RN：Detox（iOS Simulator + Android Emulator）；
  - H5：Playwright。
- 跑核心黄金路径：登录 → 进入活动 → 报名 → 支付 → 取消 → 退款。

### 10.5 调试
- **小程序**：微信开发者工具 / 支付宝小程序 IDE。
- **H5**：浏览器（同 Web 调试）。
- **RN**：Flipper / React Native DevTools；真机连接 `npx react-native start`。
- **混合调试**：单仓多终端并行，开发期通常以 H5 为主预览（最快），关键校验在小程序 + 真机 RN。

### 10.6 CI / CD

> 原则：**PR 阶段只跑 < 10min 的快速验证**，避免拖慢迭代；真机 / Release 构建放 nightly 与 release 流水线。详细 Gate 命令见 [10 §3.5](./10-技术实施细则.md)。

#### PR 流水线（每个 PR 必跑，目标 < 10min）

1. `lint` + `typecheck` + `test`（单元测试 + 组件测试）。
2. `build:weapp` + `build:h5` + `build:rn`（仅 Metro JS bundle，不打 IPA / APK）。
3. `assembleDebug`（Android Gradle Debug，约 3-5min；不跑 Release，避免 ProGuard / Hermes 编译耗时）。
4. 小程序：`miniprogram-ci` 上传体验版（仅主分支 / develop 触发），回写二维码到 PR 评论。
5. ❌ **不在 PR 阶段跑**：iOS Release、Android Release、Detox 真机 smoke。

#### Nightly 流水线（每日 02:00 自动）

1. iOS Release 构建（约 15-25min）→ TestFlight 内部测试组。
2. Android Release（含 ProGuard + Hermes + R8）→ 蒲公英内部测试组。
3. Detox iOS / Android smoke（黄金路径 P0-01 ~ P0-08）。
4. miniprogram-automator 跑小程序 P0 用例。
5. Playwright H5 smoke。
6. 任一项失败 → 钉钉告警 oncall + 阻断次日 release tag。

#### Release 流水线（打 tag `v*` 触发）

- 小程序：上传体验版 → 内部审核 → 提交审核 → 灰度发布。
- H5：构建 → 上传 OSS → 切版本号（前缀路径）。
- iOS：fastlane → TestFlight → App Store 审核；
- Android：fastlane → 蒲公英 / 应用商店；上线后用自建 OTA 灰度业务热更（仅 JS bundle，不含原生改动）。

> **真机构建成本控制**：iOS Release 单次约 20min × $0.08/min ≈ $1.6；若按 PR 频次（30/日）跑会到 $48/日；nightly 模式 = $1.6/日 × 2 平台 = $3.2/日，既保证回归覆盖，又把成本压到合理。

## 11. 性能优化

| 端 | 关注点 |
| --- | --- |
| 小程序 | 分包加载（mall 与 coach 独立分包，主包 < 2MB）；`setData` 节流；图片 webp + CDN；长列表用 `<scroll-view enable-flex>` 分片渲染 |

## 12. 已知坑与缓解

| 坑 | 缓解 |
| --- | --- |
| RN 端不支持 NutUI / DOM 组件 | 按「优先自建原子组件 + 端特化兜底」执行 |
| RN 长列表性能不足 | 使用 `@shopify/flash-list`，必要时降级原生 RecyclerView |
| 小程序包大小限制 | 分包 + 资源 CDN + 业务模块按需 import |
| 微信小程序 setData 性能瓶颈 | Taro 4 已优化（增量 diff），仍需避免大 state 频繁更新；高频用本地 ref + 节流 |
| 小程序 / RN 字体差异 | 仅使用系统字体；自定义图标用 iconfont symbol（小程序）+ vector-icons（RN），按端封装 `<Icon>` |
| 四端 navigation 心智不一 | 统一通过 `platform.navigate.*`；杜绝直接调 Taro.* / navigationRef.* |
| 真机调试 RN 卡 | 使用 New Architecture（Fabric+TurboModule），开 Hermes，预热常用页 |
| 端差异 bug 滞后发现 | CI 同步跑小程序 / H5 / RN bundle 构建 + Detox + miniprogram-automator |
| 二期高复杂度页面（直播） | 走「嵌入原生页」方案，不强行做跨端 |

## 14. 总结

- **一份代码、一种语言、一种技术栈、一种 React 心智**。
- 业务逻辑（hooks / models / 校验 / API）100% 共享；
- UI 层基于 Taro 跨端原语 + 自建原子组件 + 端特化兜底，覆盖主体业务；
- 端能力（支付、推送、扫码、地图）通过 `PlatformAdapter` + 端文件分支统一；
- 极端原生需求保留「嵌入原生页」退路，不强行跨端。
