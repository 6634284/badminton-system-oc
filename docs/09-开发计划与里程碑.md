# 09 · 开发计划与里程碑

> 交付方式假设：**1 位产品 owner + Codex 主开发**，设计、测试、运维按需支持。  
> 目标客户体量较大，所以首个可上线版本不能只是“业务能跑通”，还必须包含：  
> **高可用、高并发、回滚、审计、压测、支付稳定性**。  
> 推荐目标：**10-12 周达到试点生产版本**。

## 1. 阶段总览

```text
Sprint 0  ─ 2 周   · 架构冻结 / 数据模型冻结 / 工程骨架
M1       ─ 4 周   · 核心业务闭环（活动、报名、支付、签到、退款）
M2       ─ 3 周   · 生产能力建设（多运行时、限流、读写分离、压测）
M3       ─ 3 周   · 运营后台、财务对账、试点上线
M4       ─ 按需   · RN 强化、实时推送、复杂报表、二期能力
```

## 2. Sprint 0 — 架构与数据模型冻结（2 周）

**目标**：先把“怎么开发、怎么部署、怎么扩容、怎么回滚”定清楚，再开始写业务。

| 任务 | 主责任 | 交付 |
| --- | --- | --- |
| MVP 范围冻结 | 产品 owner + Codex | 首期只做活动、报名、支付、签到、退款、基础财务 |
| 后端技术基线 | Codex | NestJS + Fastify Adapter + PostgreSQL + Prisma + Redis + BullMQ |
| 工程骨架 | Codex | Nest monorepo、`apps + libs`、统一异常码、统一返回结构 |
| OpenAPI 流程 | Codex | Nest Swagger 导出 -> `swagger-typescript-api` 生成 SDK |
| 数据库设计冻结 | Codex | 关键表、索引、唯一约束、状态机、审计与多租户策略 |
| 部署拓扑冻结 | Codex | `client-api / admin-api / open-api / worker / scheduler` 分拆方案 |
| 压测指标冻结 | Codex | 报名洪峰、支付回调洪峰、读多写少场景目标阈值 |
| 本地联调环境 | Codex | Docker Compose 跑通 PostgreSQL、Redis、MinIO、Mock 服务 |

**里程碑**：跑通「Nest Swagger 导出 -> 前端生成 SDK -> admin-web / 小程序共用接口类型 -> 本地联调环境可用」。

## 3. M1 — 核心业务闭环（4 周）

**目标**：完成俱乐部实际可用的报名和支付主链路。

### 3.1 后端

- 认证体系：微信登录、手机号绑定、后台账号登录、JWT 刷新。
- 多租户基础：`tenant_id` 上下文、租户状态、基础 RBAC。
- 俱乐部入驻：基础信息、审核、默认配置、默认活动模板。
- 会员迁移：Excel 导入、手机号合并、会员标签、黑名单。
- 场馆与活动：球馆、场地、排期、散场活动创建 / 发布 / 取消。
- 报名抢位：Redis Lua + PostgreSQL 唯一约束 + 幂等键。
- 支付与钱包：微信支付、支付单、钱包、消费 / 退款流水。
- 签到核销：二维码签到、后台手动签到、爽约标记。
- 分享追踪：share token、邀请来源、报名转化。
- BullMQ：通知、超时关闭、补偿任务。

### 3.2 后台

- 平台后台：租户审核、租户列表、平台配置。
- 俱乐部后台：活动管理、报名列表、签到核销、会员导入、钱包流水。
- 活动模板：取消规则模板、价格模板、发布模板。
- 权限：M1 只做基础 RBAC，不做复杂可视化权限树。

### 3.3 C 端

- 微信小程序 + H5 作为首批交付重点。
- 登录、俱乐部选择、活动列表、活动详情、报名、取消、签到码、充值、流水。
- 席位变化先走短轮询。
- RN 保留构建链路，但不作为试点上线前置门槛。

### 3.4 测试

- 报名、支付、退款、钱包、支付回调全部有自动化测试。
- E2E 黄金路径：登录 -> 报名 -> 支付 -> 签到 -> 取消退款。
- 越权测试：跨租户读取、跨租户写入、伪造回调、重复提交。

### 3.5 阶段退出标准

- 业务闭环全部可跑通。
- 0 超卖，0 重复扣款，0 多租户串号。
- 内测俱乐部能独立完成发活动、导入成员、签到、退款四个操作。

## 4. M2 — 生产能力建设（3 周）

**目标**：让系统具备承接头部客户流量的基础能力，而不是只停留在“功能可用”。

| 模块 | 内容 |
| --- | --- |
| 多运行时部署 | `client-api / admin-api / open-api / worker / scheduler` 分开部署 |
| 流量治理 | API Gateway、路径分流、租户级限流、热点活动动态限流 |
| PostgreSQL | 主从读写分离、慢 SQL 监控、备份策略 |
| Redis | 高可用、热点键治理、队列资源隔离 |
| BullMQ | 队列监控、重试策略、死信与补偿流程 |
| 压测 | 热点活动报名洪峰、支付回调洪峰、只读查询高峰 |
| 发布 | 灰度发布、回滚 SOP、补偿 SOP、演练清单 |

**阶段退出标准**：

- `client-api` 在目标副本数下通过热点活动压测。
- 支付回调与普通业务流量已隔离。
- 单实例下线、单可用区故障演练有明确切流方案。
- staging 环境完成一轮完整压测与回滚演练。

## 5. M3 — 运营后台、财务对账与试点上线（3 周）

**目标**：让系统具备试点上线所需的运营、财务和恢复能力。

| 模块 | 内容 |
| --- | --- |
| 财务对账 | 对账中心、退款审批、结算单、异常流水核查 |
| 运营后台 | 审计日志、配置管理、导出中心、异常任务补偿 |
| 客户成功 | 入驻 SOP、成员导入模板、主理人操作手册、微信群迁移指引 |
| 生产发布 | staging/prod 发布流程、值班机制、告警分级 |
| 安全 | 敏感字段脱敏、支付回调验签复核、越权回归 |

**阶段退出标准**：

- 至少 1 家试点俱乐部完成真实试运行。
- 月度对账能核对充值、消费、退款、抽佣、结算。
- 故障、回滚、补偿流程均有文档与演练记录。

## 6. M4 — 按需增强

以下能力按 ROI 和真实反馈决定，不默认挤进首个生产版本：

1. RN 双端体验强化与上架。
2. WebSocket / SSE 实时席位推送。
3. 轻赛事。
4. 教练课程。
5. 复杂商城。
6. 数据仓库 / BI 深化。
7. 搜索服务深化。

## 7. 协作方式

### 7.1 分工

- **产品 owner**：优先级、业务规则、验收、资质与外部沟通。
- **Codex**：后端、后台、C 端、测试、部署脚本、文档主实现。
- **按需支持**：设计、真机测试、运营物料、发布支持。

### 7.2 节奏

- 每周至少一次里程碑回顾。
- DTO / Swagger 先改，再改前后端实现。
- 性能、可用性、安全必须和业务开发并行，不允许最后再补。

## 8. 风险登记表

| ID | 风险 | 等级 | 缓解 |
| --- | --- | --- | --- |
| R1 | 抢位超卖 | High | Redis Lua + PostgreSQL 唯一约束 + 幂等键 + 热点压测 |
| R2 | 资金账目不一致 | High | 钱包流水 + 支付/退款单 + 对账任务 + 异常告警 |
| R3 | 多租户数据串号 | High | tenant-aware repository + 审计日志 + 越权测试 |
| R4 | 支付回调被业务洪峰影响 | High | `open-api` 独立部署 + 独立限流 + 快速落库 |
| R5 | Prisma 覆盖不了热点链路 | High | 明确 Prisma + Raw SQL 双轨策略 |
| R6 | 小程序 / 支付审核延期 | Med | 沙箱联调前置，审核时间预留 |
| R7 | Redis 与队列资源争抢 | Med | seat/rate-limit/cache/queue 资源隔离 |
| R8 | 一人开发遗漏生产能力 | High | M2 专门做生产能力，不允许全部后置 |

## 9. 验收清单（首个试点生产版本）

- [ ] 接口契约完整、与实现一致。
- [ ] 报名 / 支付 / 退款 / 钱包模块自动化测试通过。
- [ ] 多租户越权测试通过。
- [ ] 热点活动压测、支付回调压测、队列恢复压测通过。
- [ ] staging / prod 发布、回滚、备份恢复至少演练一次。
- [ ] `client-api / admin-api / open-api / worker` 已分开部署。
- [ ] 监控、错误告警、审计日志可用。
- [ ] 财务对账和支付回调可追溯。
- [ ] 文档：上线手册、回滚手册、迁移手册齐全。
