# 02 · 技术架构与选型

## 0. 结论先行

当前确认采用的后端主线是：

- **NestJS**
- **Fastify Adapter**
- **PostgreSQL**
- **Prisma**
- **Redis**
- **BullMQ**
- **Swagger/OpenAPI**

同时确认以下架构原则：

1. **模块化单体开发**
2. **多运行时部署**
3. **首个生产版本就按高可用设计**
4. **普通 CRUD 走 Prisma，热点链路允许原生 SQL**

这套方案的目标不是追求某一项指标最强，而是综合满足：

- AI 代码生成质量高
- 一人持续维护成本低
- 复杂后台业务适配自然
- 小程序高峰流量可扩容
- 后续有演进空间，但不把微服务当成前置条件

## 1. 为什么选这一版

### 1.1 为什么不是 Java Spring Boot

Java 当然是稳的，但对当前项目来说问题不在于“能不能做”，而在于：

- 一个人维护 DTO / VO / Entity / Mapper / XML / service 分层，会明显更重；
- AI 生成虽然能用，但长期改动成本通常高于 NestJS；
- 当前前端已经是 TypeScript 全家桶，后端继续 TS 的整体协同收益更高。

### 1.2 为什么不是原生 Fastify + 自拼工程

原生 Fastify 很强，但当前项目是典型复杂后台业务，不是纯 API 网关。  
你会遇到：

- RBAC
- DTO 校验
- 多端 Swagger
- 支付回调
- 模块边界
- 审计日志
- 队列任务
- 后台运营能力

这些场景下，NestJS 的组织能力和 AI 生成稳定性明显更合适。

### 1.3 为什么是 PostgreSQL

这个项目中后期会持续出现以下需求：

- 时间冲突判断
- 活动复杂筛选
- JSON 扩展字段
- 财务对账
- 多条件统计
- 地理位置与范围检索

PostgreSQL 在这些方向上整体优于 MySQL，尤其是：

- `JSONB`
- GIN / B-Tree / Partial Index
- 更灵活的 SQL
- 更强的事务表达
- 需要时可用 `tstzrange`、`EXCLUDE USING GIST` 等能力

### 1.4 为什么仍然坚持单体

你现在真正要避免的不是“性能不够”，而是：

- 模块失控
- 文档和代码脱节
- AI 输出不统一
- 部署复杂度过早膨胀

所以现在不建议上微服务。  
正确做法是：

- **代码单体**
- **运行时拆分**

也就是业务模块仍在一套代码里，但运行时可以拆成：

- `client-api`
- `admin-api`
- `open-api`
- `worker`
- `scheduler`
- `ws-gateway`

## 3. 开发形态与生产形态

### 3.1 开发形态

- 后端以 **Nest monorepo** 或等价的 `apps + libs` 结构维护；
- 业务模块按领域拆分，但都在同一个仓库、同一个代码基里；
- 依赖 AI 持续生成和维护，因此目录、命名、DTO 和异常码必须强约束。

### 3.2 生产形态

生产不能把所有流量压在一个 Nest 进程上。

正式生产建议：

- `client-api`: 3+ replicas
- `admin-api`: 2+ replicas
- `open-api`: 2+ replicas
- `worker`: 2+ replicas
- `scheduler`: 2 replicas + leader election
- PostgreSQL 主库 + 至少 2 个只读副本
- Redis 高可用或 Cluster
- BullMQ 独立监控

## 4. 技术栈选型

### 4.1 后端版本基线

| 层 | 推荐 | 备注 |
| --- | --- | --- |
| Runtime | **Node.js 22 LTS** | 默认主线 |
| 语言 | **TypeScript 5.x** | `strict` 开启 |
| Web 框架 | **NestJS 11** | 统一模块、依赖注入、鉴权、Swagger、队列接入 |
| HTTP 适配器 | **@nestjs/platform-fastify** | 兼顾 NestJS 工程能力与更高吞吐 |
| ORM | **Prisma** | 默认 CRUD 与事务入口 |
| 热点 SQL | **Prisma Raw SQL** | 报名、支付、财务、复杂报表允许原生 SQL |
| 数据库 | **PostgreSQL 16** | 主业务库 |
| 缓存 | **Redis 7** | 缓存、限流、幂等、席位、会话 |
| 队列 | **BullMQ** | 延时任务、补偿、重试、导出 |
| 文档 | **@nestjs/swagger** | 自动导出 OpenAPI |
| SDK 生成 | **swagger-typescript-api** | admin-web / client-taro 统一生成 |
| 鉴权 | JWT + Refresh Token + RBAC | 平台与租户共用一套安全模型 |
| 观测 | OpenTelemetry + Prometheus + Sentry | 链路、指标、错误统一采集 |
| 测试 | Jest + Testcontainers + Supertest / app.inject | 贴近 NestJS 主流生态 |
| 压测 | k6 / autocannon | 热点活动、支付回调、只读查询高峰 |

### 4.2 Prisma 使用边界

必须明确：

- **后台 CRUD、普通详情、普通列表**：优先 Prisma
- **热点活动报名链路**：允许 raw SQL
- **钱包扣款 / 退款 / 支付回调**：允许 raw SQL
- **复杂聚合报表**：允许 raw SQL 或物化视图

原因不是 Prisma 不好，而是这些场景必须优先保证：

- 锁可控
- SQL 可控
- 性能可控
- 排障可控

### 5.1 模块内部建议结构

```text
activity/
├── activity.module.ts
├── controllers/
├── services/
├── repositories/
├── dto/
├── entities/
├── guards/
├── events/
└── tests/
```

原则：

- 每个模块对外暴露 `controller + service + module`；
- DTO、错误码、权限码跟模块一起放；
- 不提前为了“优雅”拆太多抽象层；
- 先把边界清楚，再做复用。

## 6. 前端选型

### 6.1 admin-web

- React 18
- TypeScript 5
- Vite 5
- Ant Design 5 + Ant Design Pro
- Zustand + TanStack Query

### 6.2 client-taro

- Taro 4.x
- React 语法
- 小程序 / H5 / RN 一码四端
- 平台差异通过 `PlatformAdapter` 抹平

### 6.3 前后端协同方式

- 后端由 NestJS 自动生成 Swagger/OpenAPI；
- 前端通过 `swagger-typescript-api` 生成 SDK；
- 所有接口变更必须先改 DTO / Swagger，再改前端调用。

## 7. 数据与存储选型

| 存储 | 选型 | 用途 |
| --- | --- | --- |
| 主业务库 | PostgreSQL 16 | 订单、钱包、活动、会员、支付 |
| 缓存 / 限流 / 队列 | Redis 7 | 席位、热点缓存、幂等、限流、BullMQ |
| 延时与补偿任务 | BullMQ | 订单超时关闭、补偿、导出、通知重试 |
| 对象存储 | OSS / COS / MinIO | 图片、票据、海报、导出文件 |
| 搜索 | PostgreSQL 索引起步 | 后续按复杂度引入 OpenSearch |
| BI / 报表 | ClickHouse / Doris（二期） | 经营报表、离线分析 |

## 8. 关键非功能性方案

### 8.1 并发与一致性

- 报名抢位：Redis Lua + PostgreSQL 唯一约束 + 幂等键
- 钱包扣款：事务 + 条件更新 / 乐观锁
- 支付回调：状态机 + 行锁 + 幂等
- 候补递补：延时任务 + 明确补偿路径

### 8.2 高可用

- API 无状态
- 多副本
- 多可用区
- 读写分离
- 支付回调流量与普通业务流量隔离
- 后台导出与主链路隔离

### 8.3 安全

- JWT + RBAC
- 多租户强隔离
- 审计日志
- 接口限流
- 支付回调验签
- 敏感字段脱敏与加密

## 9. 我们明确不做的事

首期不做：

- 微服务
- 过早引入复杂服务网格
- 过早引入 Kafka 作为所有异步逻辑前置依赖
- 过早做完整实物电商
- 为了技术炫技牺牲 AI 可维护性

## 10. 结论

对这个项目来说，最优解不是“最强性能语言”，而是：

- **最适合 AI 稳定生成**
- **最适合一人持续维护**
- **又足够支撑头部客户生产要求**

所以最终推荐架构就是：

**NestJS + Fastify Adapter + PostgreSQL + Prisma + Redis + BullMQ**

并且一定要配套：

**模块化单体开发 + 多运行时部署 + 首个生产版本按高可用设计**
