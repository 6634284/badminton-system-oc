# 10 · 技术实施细则

> 本文补充「怎么落地」：Taro RN 端 Hermes / Fabric 启用、OTA 灰度脚本、四端差异回归清单，以及 TypeScript 后端的模块、事务、幂等、租户、支付、高并发和测试细节。

## 1. 版本与开关基线

| 模块 | 基线 | 策略 |
| --- | --- | --- |
| C 端 | Taro 4.x + React 18 + Taro RN 输出 | 以 `client-taro` 单仓为唯一来源，四端 CI 同步构建 |
| RN Runtime | Hermes 开启，新架构 / Fabric 分阶段开启 | 先在 staging 打开，再按机型灰度；任何原生库不兼容可独立回滚 |
| OTA | 自建 OTA / CodePush 协议兼容实现 | 不依赖 App Center SaaS；只热更新 JS bundle 与静态资源 |
| 后端 | NestJS + Fastify Adapter + TypeScript 5.x + Node.js 22 LTS | 开发单仓，生产按 `client-api / admin-api / open-api / worker` 独立部署 |
| 数据 | PostgreSQL 16 + Redis 7.0 | 报名席位用 Redis Lua，资金最终以 PostgreSQL 事务与唯一约束兜底 |

## 2. Taro RN 路由接管（关闭默认导航）

> Taro 4 的 RN 输出自带一套基于 React Navigation 的默认 stack，但我们要用 [07 §6](./07-跨端前端方案.md#6-路由抽象) 中那张「单一 routes.ts → 同时驱动小程序 `app.config` 与 RN react-navigation」的方案，必须显式关闭 Taro 默认导航，由我们自己挂 `RootNav`。否则会出现两套 navigator 嵌套、返回键失效、deep link 错乱、动画双层。

### 2.1 关键原则

- **唯一来源**：`src/router/routes.ts` 是 4 端共用的路由声明。
- **小程序**：`src/app.config.ts` 由 `scripts/gen-app-config.ts` 在 `prebuild` 钩子中根据 `routes.ts` 生成；不要手工编辑。
- **RN**：根组件不挂 Taro 默认 stack；改为挂我们自己写的 `<RootNav>`（基于 `@react-navigation/native-stack` + `@react-navigation/bottom-tabs`）。
- **业务页面**：页面组件保持 100% 同源，只通过 `platform.navigate.*` 跳转，不直接依赖 `Taro.navigateTo` 或 `navigation.navigate`。

### 2.2 关闭 Taro RN 默认导航

`config/platform/rn.ts`：

```ts
export default {
  rn: {
    // 让 Taro 不自动包裹 React Navigation 容器；我们自己在 src/app.tsx 里挂
    enableMultipleClassName: true,
    runtime: { enableInnerHTML: false },
    router: {
      // Taro 4 RN：disableAutoNavigation = true 时，Taro 仅生成页面组件树，
      // 不再自动注入默认 NavigationContainer / Stack.Navigator
      disableAutoNavigation: true,
    },
  },
};
```

> 字段名以当前 Taro 版本文档为准；项目用 Taro 4.0+ 的，`router.disableAutoNavigation` 是公开开关。若版本变更，等价做法是在 `src/app.tsx` 里覆写 Taro 注入的 `_PageWrapper` 直接 return children。

### 2.3 RN 入口：自挂 RootNav

`src/app.tsx`：

```tsx
import { Component, PropsWithChildren } from 'react';
import { QueryClient, QueryClientProvider } from '@tanstack/react-query';
import { registerPlatform } from '@/platform/adapter';
import { RootNav } from '@/router/nav.rn';                   // RN 端导航容器
import 'react-native-gesture-handler';                       // RN 必须最早 import
import { GestureHandlerRootView } from 'react-native-gesture-handler';
import { SafeAreaProvider } from 'react-native-safe-area-context';
import { NavigationContainer } from '@react-navigation/native';
import { navigationRef, linking } from '@/router/nav.rn';
import platformRn from '@/platform/adapter.rn';

const qc = new QueryClient();

export default class App extends Component<PropsWithChildren> {
  componentWillMount() {
    if (process.env.TARO_ENV === 'rn') {
      registerPlatform(platformRn);
    }
  }

  render() {
    if (process.env.TARO_ENV === 'rn') {
      return (
        <GestureHandlerRootView style={{ flex: 1 }}>
          <SafeAreaProvider>
            <QueryClientProvider client={qc}>
              <NavigationContainer ref={navigationRef} linking={linking}>
                <RootNav />
              </NavigationContainer>
            </QueryClientProvider>
          </SafeAreaProvider>
        </GestureHandlerRootView>
      );
    }
    // 小程序 / H5：Taro 接管，children 即 Taro 渲染的 page 组件
    return (
      <QueryClientProvider client={qc}>
        {this.props.children}
      </QueryClientProvider>
    );
  }
}
```

### 2.4 单一 routes.ts 同时驱动两端

```ts
// src/router/routes.ts
export interface AppRoute {
  name: string;
  path: string;                    // 小程序 page path，也是 H5 path
  rnDeepLink?: string;             // RN linking config 的 deep link 模板，含 :id 占位
  component: React.ComponentType<any>;
  tab?: boolean;
  meta?: { title?: string; auth?: boolean };
}

export const routes: AppRoute[] = [
  { name: 'Home',           path: '/pages/home/index',
    rnDeepLink: '/home',           component: HomePage,           tab: true },
  { name: 'ActivityList',   path: '/pages/activity/list/index',
    rnDeepLink: '/activities',     component: ActivityListPage,   tab: true },
  { name: 'ActivityDetail', path: '/pages/activity/detail/index',
    rnDeepLink: '/activities/:id', component: ActivityDetailPage },
  // ...
];
```

`src/router/nav.rn.ts`：

```ts
import { createNativeStackNavigator } from '@react-navigation/native-stack';
import { createBottomTabNavigator } from '@react-navigation/bottom-tabs';
import { createNavigationContainerRef, LinkingOptions } from '@react-navigation/native';
import { routes } from './routes';

const Stack = createNativeStackNavigator();
const Tab = createBottomTabNavigator();

export const navigationRef = createNavigationContainerRef();

export const linking: LinkingOptions<any> = {
  prefixes: ['badminton://', 'https://m.example.com'],
  config: {
    screens: Object.fromEntries(
      routes.filter(r => r.rnDeepLink).map(r => [r.name, r.rnDeepLink!]),
    ),
  },
};

const TabsNav = () => (
  <Tab.Navigator>
    {routes.filter(r => r.tab).map(r => (
      <Tab.Screen key={r.name} name={r.name} component={r.component}
        options={{ title: r.meta?.title }} />
    ))}
  </Tab.Navigator>
);

export const RootNav = () => (
  <Stack.Navigator screenOptions={{ headerShown: false }}>
    <Stack.Screen name="Tabs" component={TabsNav} />
    {routes.filter(r => !r.tab).map(r => (
      <Stack.Screen key={r.name} name={r.name} component={r.component} />
    ))}
  </Stack.Navigator>
);
```

`scripts/gen-app-config.ts`（小程序 prebuild 钩子）：

```ts
import { routes } from '../src/router/routes';
import fs from 'node:fs';

const pages = routes.map(r => r.path.replace(/^\//, ''));
```


### 2.5 PlatformAdapter 导航实现

`src/platform/adapter.rn.ts`：

```ts
import { navigationRef } from '@/router/nav.rn';

const navigate: PlatformAdapter['navigate'] = {
  push: (name, params) => navigationRef.current?.navigate(name as never, params as never),
  back: () => navigationRef.current?.canGoBack() && navigationRef.current.goBack(),
  redirect: (name) => navigationRef.current?.reset({
    index: 0, routes: [{ name: name as never }],
  }),
};
```

`src/platform/adapter.weapp.ts`：

```ts
import Taro from '@tarojs/taro';
import { routes } from '@/router/routes';

const findPath = (name: string) => routes.find(r => r.name === name)?.path
  ?? (() => { throw new Error(`route ${name} not found`); })();

const navigate: PlatformAdapter['navigate'] = {
  push: (name, params) => Taro.navigateTo({ url: `${findPath(name)}?${qs(params)}` }),
  back: () => Taro.navigateBack({ delta: 1 }),
  redirect: (name) => Taro.redirectTo({ url: findPath(name) }),
};
```

### 2.6 验收清单

- [ ] `taro build --type rn` 后启动 RN，**根视图只有一层 NavigationContainer**（用 React Navigation devtools 验证）。
- [ ] 物理返回键 / 手势返回 / `platform.navigate.back()` 行为一致，无两层栈。
- [ ] 深链 `badminton://activities/123` 在 RN 端能直接落到 ActivityDetail 并回退到 Home。
- [ ] 小程序端 `app.config.ts` 是脚本生成的，git diff 后无意外手工修改。
- [ ] 路由表新增一条记录，分别在小程序、RN、H5 三端各跑一遍 smoke。

### 2.7 退路

如 Taro 版本暂不支持 `disableAutoNavigation`：

- 短期方案：在 `src/app.tsx` 内通过 React 高阶组件包裹 Taro 注入的 PageWrapper，仅在 `process.env.TARO_ENV === 'rn'` 时直接 return children，让 Taro 默认 navigator 不工作；
- 中期：升级 Taro 4 到支持版本；
- 长期：若仍受限，最坏情况就是把 RN 端独立成 `client-rn-shell`（壳工程），src/ 仍 100% 共用 —— 但这是 fallback，不应作为默认设计。

## 3. Taro RN：Hermes / Fabric 启用步骤

React Native 新项目中 Hermes 已是默认 JS 引擎；React Native 0.76 起新架构默认开启。这里仍显式写出开关和验证步骤，是为了适配 Taro RN 模板、旧工程迁移、CI 可观测和灰度回滚。

### 3.1 前置检查

- `client-taro` 必须能跑通 `pnpm build:rn`。
- RN 端依赖不得直接从业务组件 import DOM-only 包，如 NutUI。
- `adapter.rn.ts` 中使用的原生库必须逐一确认支持当前 RN 版本和新架构。
- 先建立两条构建产物：`rn-oldarch-hermes` 与 `rn-newarch-hermes`，便于快速回退。

### 3.2 Android

`android/gradle.properties`：

```properties
hermesEnabled=true
newArchEnabled=true
android.useAndroidX=true
```

构建命令：

```bash
cd client-taro
pnpm build:rn
cd android
./gradlew clean
./gradlew :app:assembleRelease -PhermesEnabled=true -PnewArchEnabled=true
```

验证项：

- Release 包启动后执行 `!!global.HermesInternal === true`。
- 启动日志中无 `TurboModule` / `Fabric` 初始化异常。
- `ActivityDetail`、长列表、支付 SDK、扫码、图片上传都能进入页面并完成 smoke。

### 3.3 iOS

`ios/Podfile` 按 Taro / RN 模板保留原生成结构，在 `use_react_native!` 附近打开 Hermes 和新架构。不同 RN 模板字段名会有差异，以生成文件为准：

```ruby
ENV['RCT_NEW_ARCH_ENABLED'] = '1'

use_react_native!(
  :path => config[:reactNativePath],
  :hermes_enabled => true,
  # 若模板暴露 fabric_enabled / flags，则与 new arch 一起开启
)
```

安装与构建：

```bash
cd client-taro
pnpm build:rn
cd ios
RCT_NEW_ARCH_ENABLED=1 bundle exec pod install
xcodebuild \
  -workspace Badminton.xcworkspace \
  -scheme Badminton \
  -configuration Release \
  -sdk iphonesimulator \
  -derivedDataPath build
```

验证项：

- Xcode 启动日志无 `RCTTurboModuleManager` / `Fabric` 相关 crash。
- `global.nativeFabricUIManager != null` 或 RN DevTools 能看到 Fabric renderer。
- iOS 低内存机型至少跑 20 分钟页面切换和长列表滚动。

### 3.4 回滚策略

- 新架构问题：保留 `newArchEnabled=false` / `RCT_NEW_ARCH_ENABLED=0` 构建参数，打老架构包。
- Hermes 问题：仅在定位到 JS 引擎兼容问题时临时关闭；关闭 Hermes 会影响启动性能，不作为长期方案。
- 原生库问题：优先隔离到 `native/` 和 `adapter.rn.ts`，不得让业务组件直接依赖问题库。

### 3.5 CI Gate

> 与 [07 §10.6](./07-跨端前端方案.md#106-ci--cd) 对齐：PR 阶段只跑 < 10min 的快速验证，iOS Release 与 Detox 真机回归放 nightly。

#### PR Gate（必须通过，目标 < 10min）

```bash
pnpm lint
pnpm typecheck
pnpm test
pnpm build:weapp
pnpm build:h5
pnpm build:rn
cd android && ./gradlew :app:assembleDebug -PhermesEnabled=true -PnewArchEnabled=true
cd ios && RCT_NEW_ARCH_ENABLED=1 bundle exec pod install
```

#### Nightly Gate（每日 02:00 自动）

```bash
# Android Release：含 ProGuard + Hermes + R8
cd android && ./gradlew :app:assembleRelease -PhermesEnabled=true -PnewArchEnabled=true

# iOS Release：fastlane 走 TestFlight 内部测试组
cd ios && fastlane beta

# 三端 E2E
pnpm e2e:weapp        # miniprogram-automator
pnpm e2e:h5           # Playwright
pnpm e2e:rn           # Detox（iOS + Android）
```

任一失败 → 钉钉告警 oncall + 阻断次日 release tag。

## 4. OTA / CodePush 灰度方案

### 4.1 原则

App Center / CodePush SaaS 已在 2025-03-31 退休，不作为依赖项。项目采用 **自建 OTA 服务 + CodePush 类似协议**：

- 只发布 JS bundle、source map、静态资源。
- 不热更新原生模块、权限、支付 SDK、Info.plist、AndroidManifest、Pod / Gradle 依赖。
- 每个 OTA 包绑定 `targetBinaryVersion`、`platform`、`channel`、`minNativeBuild`。
- 灰度按 `user_id` / `device_id` 一致性 hash，避免用户反复进出灰度组。
- 支付、报名、退款链路出问题必须支持一键回滚。

### 4.2 包结构

```
ota/
├── releases/
│   └── rn-ios-production-20260520.1.zip
└── manifest.json
```

`manifest.json` 示例：

```json
{
  "label": "rn-ios-production-20260520.1",
  "platform": "ios",
  "channel": "production",
  "targetBinaryVersion": "1.0.x",
  "minNativeBuild": 100,
  "rollout": 10,
  "mandatory": false,
  "bundleUrl": "https://cdn.example.com/ota/rn-ios-production-20260520.1.zip",
  "bundleHash": "sha256:...",
  "createdAt": "2026-05-20T12:00:00Z",
  "notes": "fix activity detail scroll restore"
}
```

### 4.3 package.json 脚本

```jsonc
{
  "scripts": {
    "ota:bundle:ios": "react-native bundle --platform ios --dev false --entry-file index.js --bundle-output build/ota/ios/main.jsbundle --assets-dest build/ota/ios/assets --sourcemap-output build/ota/ios/main.jsbundle.map",
    "ota:bundle:android": "react-native bundle --platform android --dev false --entry-file index.js --bundle-output build/ota/android/index.android.bundle --assets-dest build/ota/android/assets --sourcemap-output build/ota/android/index.android.bundle.map",
    "ota:release": "node scripts/ota/release.mjs",
    "ota:promote": "node scripts/ota/promote.mjs",
    "ota:rollback": "node scripts/ota/rollback.mjs"
  }
}
```

### 4.4 发布脚本参数

```bash
# staging 全量
pnpm ota:release \
  --platform ios \
  --channel staging \
  --target-binary 1.0.x \
  --rollout 100 \
  --notes "activity detail smoke"

# production 5% 灰度
pnpm ota:promote \
  --from staging \
  --to production \
  --platform ios \
  --rollout 5

# 扩到 20%
pnpm ota:promote \
  --to production \
  --platform ios \
  --label rn-ios-production-20260520.1 \
  --rollout 20

# 回滚到上一稳定包
pnpm ota:rollback \
  --platform ios \
  --channel production \
  --to-label rn-ios-production-20260518.2
```

### 4.5 服务端灰度判定

```java
boolean hitRollout(String stableId, int rollout) {
    int bucket = Math.floorMod(Murmur3.hash32(stableId), 100);
    return bucket < rollout;
}
```

灰度条件：

- `platform` 匹配。
- `appVersion` 命中 `targetBinaryVersion`。
- `nativeBuild >= minNativeBuild`。
- `tenant_id` 不在黑名单。
- `user_id/device_id` 命中 rollout bucket。
- 最近 10 分钟该 label 的 crash rate、支付失败率、报名失败率未超过阈值。

### 4.6 客户端检查节奏

- App 启动 5 秒后检查一次。
- 从后台回前台超过 30 分钟检查一次。
- 活动报名 / 支付进行中不安装更新，只下载。
- 安装时机：下次冷启动，或用户离开核心交易页后提示重启。

### 5.2 端能力回归

| 能力 | 风险 | 回归点 |
| --- | --- | --- |
| 路由 | 小程序 path 与 RN route name 不一致 | 所有 `platform.navigate.*` 参数统一通过路由表生成 |
| Storage | RN MMKV / 小程序 Storage 序列化差异 | token、tenant、query cache 冷启动恢复 |
| 支付 | SDK 参数格式差异 | 微信小程序 `requestPayment`、RN 微信 / 支付宝 SDK、H5 跳转兜底 |
| 分享 | 小程序 open-type 与 RN share sheet 差异 | share token 必须能归因到报名 |
| 扫码 | 权限、相机、模拟器能力差异 | 拒绝权限、再次授权、扫码失败兜底输入 |
| 长列表 | RN FlatList / FlashList 与小程序 scroll-view 差异 | 加载更多、下拉刷新、返回保位、空状态 |
| 表单 | 键盘遮挡、中文输入法、数字输入 | 价格、手机号、验证码、备注输入 |
| Safe Area | iPhone 刘海屏 / Android 导航栏 | 底部报名栏不被遮挡 |
| 图片 | OSS 图片、WebP、缓存策略 | 活动封面、头像、海报二维码显示 |
| 时间 | 时区、日期格式、跨天活动 | 报名截止、取消退款规则、排期日期 |
| WebSocket / 轮询 | 后台切前台、断线重连 | 席位变化、候补变化、活动取消 |
| 暗色模式 | token 映射不一致 | 文字对比度、二维码、底部栏 |

### 5.3 自动化分层

- `unit`：hooks、models、validators、price/date/tenant utils。
- `component`：ActivityCard、SeatGrid、FixedBar、WalletHeader。
- `e2e-weapp`：miniprogram-automator 跑 P0-01 到 P0-08。
- `e2e-h5`：Playwright 跑分享落地页、登录跳转、报名预览。
- `e2e-rn`：Detox 跑 iOS / Android smoke，重点覆盖支付前后状态和 OTA。

```text
    │   ├── redis/
    │   ├── queue/
    │   ├── storage/
    │   └── messaging/
    └── shared/
        ├── auth/
        ├── errors/
        ├── result/
        ├── audit/
        └── context/
```

原则：

- 代码共享，但运行时职责分离。
- `client-api`、`admin-api`、`open-api`、`worker` 必须能独立启动、独立配置、独立扩容。
- 公共模块只放稳定能力，不把业务逻辑都塞进 `shared`。

### 6.2 Bootstrap 基线

所有 API 运行时统一使用 NestJS + Fastify Adapter：

```ts
async function bootstrap() {
  const app = await NestFactory.create<NestFastifyApplication>(
    ClientApiModule,
    new FastifyAdapter({ logger: false }),
  );

  app.useGlobalPipes(new ValidationPipe({
    whitelist: true,
    transform: true,
    forbidNonWhitelisted: true,
  }));

  app.useGlobalFilters(new AppExceptionFilter());
  app.useGlobalInterceptors(
    new TraceInterceptor(),
    new ResponseTransformInterceptor(),
    new AuditInterceptor(),
  );

  const config = new DocumentBuilder()
    .setTitle('Badminton Client API')
    .setVersion('1.0')
    .addBearerAuth()
    .build();

  SwaggerModule.setup('/docs', app, SwaggerModule.createDocument(app, config));

  await app.listen(3000, '0.0.0.0');
}
```

必须遵守：

- API 容器内只跑 1 个 Node 主进程；
- 不在容器内用 PM2 cluster 扩容；
- 统一由多副本 + HPA 扩容。

### 6.3 Nest 组件职责划分

| 组件 | 作用 |
| --- | --- |
| `Guard` | JWT、权限、租户访问控制 |
| `Pipe` | DTO 校验和参数转换 |
| `Interceptor` | trace、响应包装、审计、缓存、超时 |
| `Filter` | 异常统一收口 |
| `Service` | 业务编排和事务边界 |
| `Repository` | Prisma / Raw SQL 数据访问 |

### 6.4 Swagger / OpenAPI 流程

- DTO 使用 `class-validator` + `class-transformer` + `@nestjs/swagger` 装饰器。
- 每次后端变更后，在 CI 里导出 OpenAPI JSON。
- 前端通过 `swagger-typescript-api` 生成 SDK。
- 后端 DTO 是事实来源，禁止手工维护多份接口定义。

### 6.5 Prisma 使用边界

默认原则：

- 后台 CRUD、简单列表、普通详情：优先 Prisma。
- 热点活动报名、钱包扣款、支付回调、复杂财务报表：允许 Raw SQL。

约束：

- 所有 Raw SQL 都必须落在 `repository` 层；
- 任何包含锁、批量更新、复杂聚合的逻辑，都要优先保证 SQL 可控；
- 不允许控制器里直接写 SQL。

### 6.6 PrismaService 与连接池

- `PrismaService` 由 Nest 全局单例提供。
- 连接池大小必须按：
  - 单实例连接数
  - 副本数
  - PostgreSQL 最大连接数
 统一反推。

禁止：

- 副本盲目扩容但不调整 DB 连接池；
- 在任务进程和 API 进程里使用不同的无约束连接策略。

### 6.7 多租户隔离

- `TenantGuard` 或等价 Guard 从 JWT 与 `X-Tenant-Id` 解析租户上下文。
- C 端传来的 `X-Tenant-Id` 只表示“当前访问意图”，不能直接信任，必须验证用户和租户关系。
- B 端从 JWT 中读取当前租户；平台账号切租户必须落审计日志。
- 所有 repository 默认带 `tenant_id` 过滤；系统越权查询必须显式记录 reason。

### 6.8 幂等实现

- 所有 `POST / PATCH / DELETE` 写接口必须带 `Idempotency-Key`。
- Redis key：`idem:{tenant_id}:{user_id}:{method}:{path}:{key}`。
- 状态：
  - `PROCESSING`
  - `SUCCESS`
  - `FAILED_RETRYABLE`
- 成功响应缓存 60-300 秒。
- 支付回调类幂等以 **业务单号 + PostgreSQL 行锁/唯一约束** 为准，不只依赖 Redis。

### 6.9 PostgreSQL 事务策略

#### 普通事务

- 默认走 Prisma `$transaction`。
- 一个事务只覆盖必要表，避免过大事务。

#### 热点事务

以下场景允许直接使用 Raw SQL + PostgreSQL 特性：

- `SELECT ... FOR UPDATE`
- `FOR UPDATE SKIP LOCKED`
- 条件更新
- 批量结算
- 对账修复

需要时可用：

- `pg_advisory_lock`
- `partial index`
- `jsonb_path_ops`
- `materialized view`

但首版预约模型仍建议保留“排期块 + 唯一约束”的简单设计，不要一开始就把 `tstzrange` 和排斥约束引入全系统。

### 6.10 报名抢位事务

执行顺序：

1. 网关限流 + 租户级限流 + 活动级动态限流。
2. 校验幂等。
3. 读活动并校验状态、报名窗口、黑名单、取消规则。
4. Redis Lua 原子扣减 `left`，并记录用户去重。
5. 开启 PostgreSQL 事务。
6. 写 `activity_registrations`，`active_user_key` 防重复活跃报名。
7. 写 `registration_participants`，`seat_no` 防重复席位。
8. 钱包扣款或创建支付单。
9. 写 outbox 事件。
10. 提交事务后异步通知 / 刷新缓存 / 推送消息。
11. 任一步失败释放 Redis 席位；释放失败写补偿任务。

热点活动保护要求：

- 活动级令牌桶阈值可动态调整。
- 热点活动详情页的席位快照要缓存并带 `version`。
- 报名按钮前置灰态和排队提示，减少无效打点。

### 6.11 钱包扣款与余额安全

余额扣减仍使用条件更新与乐观控制：

```sql
UPDATE wallets
SET
  gift_balance = gift_balance - $1,
  cash_balance = cash_balance - $2,
  version = version + 1,
  updated_at = now()
WHERE id = $3
  AND tenant_id = $4
  AND version = $5
  AND gift_balance >= $1
  AND cash_balance >= $2;
```

规则：

- 更新行数必须等于 1，否则最多重试 3 次。
- 每次扣款必须同事务写 `wallet_transactions`。
- 钱包流水不可更新、不可删除，只允许冲正。
- 钱包与支付状态推进必须可审计、可回放。

### 6.12 支付回调隔离

- `open-api` 独立部署，不能和普通业务流量共享同一条伸缩曲线。
- `payment_orders.out_trade_no` 唯一。
- 回调入口先验签，再锁支付单。
- 若已 `paid`，直接返回成功。
- 状态推进必须在事务内处理业务单：
  - `recharge`：钱包入账 + 流水
  - `activity`：报名支付成功 + 席位确认
  - `mall`：订单支付成功 + 库存扣减
- 原始回调报文脱敏后写 `notify_payload`。
- 回调洪峰场景下允许先落库、后异步扇出，不把耗时逻辑堵在 HTTP 回调里。

### 6.13 BullMQ、Outbox 与 Kafka 分工

`outbox_events` 建议字段：

```sql
id, tenant_id, event_type, aggregate_type, aggregate_id,
payload_json, status, retry_count, next_retry_at, created_at, updated_at
```

规则：

- 业务事务内只写 PostgreSQL 和 outbox。
- `worker` 扫描 `pending` outbox，并投递到：
  - BullMQ：延时关闭、重试、补偿、导出
  - Kafka / MQ：通知广播、异步同步、埋点、报表流水
- 发送成功标记 `sent`；失败指数退避。
- 所有消费者必须业务幂等，允许至少一次投递，不允许重复扣款或重复入账。

### 6.14 读写分离与查询策略

- 报名、支付、钱包、退款核心链路全部走主库。
- 列表、报表、后台审计等读流量优先走只读副本。
- 用户刚完成写操作后的关键确认页，必须走主库或强一致缓存。
- 大报表走异步导出，不走同步接口。

### 6.15 高可用约束

- API 服务无状态，多副本，多可用区。
- `scheduler` 多副本部署，但通过 leader election 或 DB/Redis 锁保证同一任务只跑一次。
- `worker` 消费失败要支持重试、退避、死信记录和人工补偿。
- PostgreSQL、Redis、BullMQ 任何一个都不能成为单点。
- 单实例退出、单 Pod 重启、单 AZ 故障不能导致核心链路全站不可用。

## 7. 关键上线阈值

- P0 接口错误率 < 0.1%。
- `client-api` 集群在首个生产目标配置下稳定承载 **1500+ QPS** 综合流量。
- 热点活动报名链路稳定承载 **800+ QPS/活动**，无超卖。
- 报名写路径 P99 < 400ms；缓存命中的活动详情 P99 < 120ms。
- Redis Lua 执行 P99 < 20ms。
- 支付回调重复投递 1000 次，不重复入账。
- 钱包并发扣款压测后余额不为负，流水和余额快照一致。
- 单实例下线或单 Pod 重启不影响整体服务可用性。
- staging 完成至少一次热点活动压测、一次回调风暴压测、一次回滚演练后才允许生产灰度。
- 四端 P0 用例全绿，RN OTA 5% 灰度 30 分钟 crash-free session 达标后再扩量。
